# Projection Manifest Source Hash Is Prompt Name Hash

## Finding

`ProjectionPromptSourceHash` is populated with `SHA256(projection.ProjectionPromptName)` instead of the generated prompt template's source hash.

Relevant code:

- `src/CommandCenter.Roadmap.CLI/ProjectionCache.cs`
- `src/CommandCenter.Roadmap.CLI/ProjectionManifest.cs`
- `src/CommandCenter.Roadmap.CLI/ProjectionManifestStore.cs`
- `docs/prompt-architecture.md`

The manifest column is named `Projection Prompt Source Hash`, and the prompt architecture documents `SourceHash` as the build-time hash of the raw `.prompt` template text. Hashing the prompt name does not detect template changes because the name remains stable across prompt edits.

## Impact

Projection provenance is misleading. A manifest row appears to pin a projection to the exact prompt source that produced it, but it only pins the prompt identifier. If a projection prompt changes while the Project Context remains the same, existing projection files can look fresh even though they were generated by an older prompt template.

This weakens auditability and stale projection detection for roadmap transitions.

## Root Cause

`RoadmapPromptCatalog` exposes render methods but not prompt metadata. `ProjectionCache` therefore only has the projection prompt name and computes a placeholder hash from that name.

The generated prompt catalog already exposes source hashes for prompt classes, but the roadmap CLI has no typed metadata bridge for them.

## Proposal

Promote projection prompt metadata into the roadmap prompt catalog.

Suggested shape:

```csharp
internal sealed record PromptTemplateMetadata(
    string PromptName,
    string PromptType,
    string SourceHash);
```

Add methods:

```csharp
internal static PromptTemplateMetadata GetProjectionMetadata(string projectionPromptName);
internal static PromptTemplateMetadata GetRuntimeMetadata(string runtimePromptName);
```

For each projection prompt, return the generated catalog type and its `SourceHash`, for example:

```csharp
"ProjectionForSelectNextEpic" =>
    new(
        "ProjectionForSelectNextEpic",
        typeof(CommandCenter.Core.Prompts.Projections.ProjectionForSelectNextEpic).FullName!,
        CommandCenter.Core.Prompts.Projections.ProjectionForSelectNextEpic.SourceHash)
```

Then `ProjectionCache` should write:

- projection prompt name
- projection prompt type
- projection prompt source hash
- project context hash
- generated projection hash

Staleness should account for prompt source drift:

- If the existing manifest entry's `ProjectionPromptSourceHash` differs from current metadata, mark stale.
- If stale policy is `Block`, block and instruct the user to regenerate the projection.
- If a projection is regenerated, persist the current source hash.

This preserves the existing manifest model while making the provenance field truthful and actionable.

## Acceptance Criteria

- `ProjectionPromptSourceHash` equals the generated prompt class `SourceHash`, not a hash of the prompt name.
- A changed projection prompt source hash marks an existing projection stale.
- Stale prompt-source projections are blocked for contracts with `StaleProjectionPolicy.Block`.
- The manifest round-trip test covers source hash persistence.
- A regression test proves two different source hashes for the same prompt name are treated as stale.
