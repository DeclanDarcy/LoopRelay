## Executive Summary

This is a static, source-level architectural pass over Command Center post-MVP. Runtime behavior, performance, render-frequency, and live-payload measurement were deliberately DEFERRED to a second, instrumented pass; every finding below is grounded in file:line evidence or in interpretation explicitly derived from it, and nothing here should be read as a measured runtime claim.

**Overall verdict: structurally sound foundations with a consistent, fixable class of seam-level debt — net "mixed-to-adequate."** Across twelve dimensions the ratings cluster at `strong` (1: static dependency graph), `adequate` (2: UI authority boundary, backend abstraction), `mixed` (8), and `at-risk` (1: mock fidelity & drift prevention). No dimension is rated "broken," and the load-bearing skeletons — the acyclic backend project DAG on a genuine Core base, the directional UI module graph with a single Tauri transport seam, the per-domain DI composition root, and the design-system/status-presentation layers — are genuinely well-designed and should be left alone. The debt is concentrated and thematic rather than diffuse, which is the optimistic reading: a small number of structural decisions, fixed once, retire findings across many dimensions.

The 5 highest-leverage cross-cutting themes:

1. **The contract is hand-maintained in three-to-four parallel copies (C# DTO → Rust shell struct → TS type → dev mock) with no generator, no contract test, and a reportedly broken `tsc`.** This single root cause produces the most severe findings in four separate dimensions: the shell silently strips `decisionSessionSummary` from both repository projections (SHELL-1/CONTRACT-1, both verified), the GET vs POST `/decisions/context` shape collapse that crashes `DecisionLifecycleTab.tsx:268` (PROJ-3, CONTRACT-1, MOCK-1, TEST-1, all verified), the certification contract that invents `severity`/`message` fields the backend never emits (CONTRACT-3), and the dev mock that "validates the UI against itself." Establishing one source of truth (codegen from the C# DTOs, or at minimum a CI contract test diffing live JSON against the TS types) is the single most valuable structural move available.

2. **The Tauri shell is not the uniform passive proxy it is assumed to be.** ~140 commands correctly forward opaque `serde_json::Value` and cannot drift; ~26 deserialize through hand-mirrored typed structs that act as a lossy filter, and `push_execution` reclassifies HTTP 409 as an `Ok` result — encoding domain outcome semantics in the shell. The two bridge strategies coexist in one file with no principled boundary. Converging on `Value` passthrough for all backend-projection responses deletes ~250 lines of mirror structs and eliminates the silent-drop bug class outright.

3. **Resilience and authority exist but are placed at the wrong seam.** The backend has no global exception handler — every endpoint re-implements a divergent inline try/catch, so `IOException` from locked snapshots escapes as raw 500s on the largest endpoint groups (RES-1, verified). The UI has exactly one root-only error boundary while all tabs mount simultaneously, so any render throw blanks the whole window (RES-2, COMP-2, verified). And `DecisionProjectionService` — a type named like a read-model — has quietly become the execution gate that sets `LaunchBlocked` via brittle lowercase substring matching (PROJ-2, verified). In each case the fix is to give an unowned cross-cutting concern a single explicit home.

4. **App.tsx is a 2184-line god component, and React owns more than presentation.** It holds ~25 useState, ~29 data hooks, 24 effects, and ~21 cross-domain handlers (COMP-1, verified), with execution state triplicated across three caches reconciled by hand-copied field lists (STATE-1, verified). Below it, the UI re-derives domain semantics the backend already owns: severity→tone via six divergent keyword tables (UIAUTH-1, EXPL-2), operational-context markdown re-parsed in parallel with the backend parser (UIAUTH-2), a client-side "Healthy" verdict fallback (UIAUTH-3), and reasoning/recovery thresholds recomputed in TypeScript (EXPL-1). These are duplications of authority, not net-new logic — bounded, but drift-prone.

5. **Pervasive copy-paste at the abstraction layer on both tiers.** The frontend has ~32 repository-keyed read hooks reproducing an identical fetch/guard/effect template (HOOK-1, verified), one of which has already silently diverged by adding a stale-response guard the others lack (HOOK-3). The backend has 27 nullable-optional DI dependencies that exist only for hand-constructed tests and misstate the production contract (ABS-1), plus a JSON-options factory duplicated four times with visible drift (ABS-4). A single shared `useRepositoryResource` factory and required (non-nullable) DI both remove ~1000+ lines and a class of divergence.

The "Projection" naming collision (read-model DTO vs markdown renderer vs execution authority, PROJ-1) and the test suite's structural blindness to contract drift (every UI test sources from the mock; the only "endpoint" test checks routing, not shape — TEST-1/2, MOCK-2/3, verified) are the connective tissue that lets all of the above persist undetected.

## Confidence Assessment

The four tiers below are kept deliberately separate. Where a finding carries a `verification` object with `evidenceHolds === true`, its evidence was independently re-checked against the source; those are the most solid. No finding in this audit was downgraded — there are zero `evidenceHolds === false` results, so nothing is excluded from the ranked themes above on verification grounds.

**(a) Observed evidence — solid.** These are file:line facts, independently re-verified, and safe to act on as ground truth:
- The backend project graph is an acyclic DAG on a real Core base; the UI module graph is directionally layered with `api/tauri.ts` as the sole Tauri coupling point (extracted from all 9 `.csproj` files and UI import-edge counts).
- The shell omits `decisionSessionSummary` from both Rust projection structs while C# emits it and TS expects it — verified literally on all three sides plus the UI `?? EMPTY` mask (SHELL-1/CONTRACT-1).
- GET `/decisions/context` returns flat `DecisionContext` while the UI types/mock expect wrapped `DecisionContextSnapshot`, crashing `DecisionLifecycleTab.tsx:268` — verified across endpoint, service, TS type, mock, and the live crash line (PROJ-3, CONTRACT-1, MOCK-1, TEST-1).
- Zero `System.Text.Json` contract attributes repo-wide; only a global enum converter; hand-written TS types; `tsc -b` reported broken (CONTRACT-2, verified — with minor corrected line counts).
- `DecisionProjectionService` conflicts feed `validationErrors` → `LaunchBlocked` via hardcoded `DirectivePrefixes`/keyword substring matching (PROJ-2, verified).
- No global backend exception handler; divergent per-file catch sets; only Artifacts/Continuity/Repositories catch `IOException` (RES-1, verified).
- One root-only React error boundary; six of seven tabs co-mounted, hidden via CSS only (RES-2, COMP-2, verified).
- App.tsx is 2184 lines with the cited hook/state/effect/handler counts; DecisionLifecycleTab is 882 lines with 9 internal eager-fetch hooks (COMP-1/COMP-4, verified); execution state triplicated with hand-copied field lists (STATE-1, verified).
- ~32 hooks are 51-line copies of one template; `useDecisionInfluence` alone adds a request-sequence guard (HOOK-1/HOOK-3, verified).
- Every UI test sources from `devTauriMock` or hand-typed objects; `BackendEndpointDispositionTests` checks routing disposition, not response shape (MOCK-2/3, TEST-1/2, verified).

Minor evidence corrections surfaced during verification (none affect any conclusion): several field counts were off by one (`ExecutionSessionSummary` 27 not 28; `OperationalContextProjection` 18 not 20; TS line counts 1365/452/355); `devTauriMock.ts` is 4963 lines, not 5179; a handful of file paths were under-qualified (e.g. the mock is at `src/CommandCenter.UI/src/devTauriMock.ts`, not under `src/test/`). These are catalogued in the per-finding `correctedEvidence` fields.

**(b) Interpretation — reasoned from the evidence, not itself observed.** Treat these as well-supported architectural judgments rather than facts: that the shell's typed structs constitute a lossy "third copy" of the contract (vs merely being unused waypoints); that the nullable-optional DI params are "poor-man's-DI bleeding into production"; that `DecisionProjectionService` is "authority leakage" because of its name/position (its position as an authority is verified; the *characterization* as leakage is interpretation); that the six `toneFrom*` tables represent a single forked presentation policy; and that App.tsx's god-component shape is the structural *cause* (not just correlate) of the wide tab-prop contracts. These are coherent and consistently argued, but a reasonable architect could frame some differently.

**(c) Recommendation — proposed direction, not yet validated by implementation.** Codegen/contract-test for the wire boundary; collapsing the shell to uniform `Value` passthrough; a global `IExceptionHandler`; per-tab error boundaries + conditional mounting; a shared `useRepositoryResource` factory and `useAsyncAction`; required (non-nullable) DI sourced from `Program.CreateApp`; per-feature controller hooks to decompose App.tsx; a shared tone module; and emitting backend-owned severity/health/recommendation tiers so the UI stops recomputing them. These are the audit's opinion on the highest-leverage fixes; their risk notes (several rated medium-to-high churn) are themselves estimates.

**(d) Speculation — explicitly flagged as unconfirmed.** Carry these forward only as hypotheses for the runtime pass: that the nullable-optional DI exists specifically for test construction; that `latestPushAttemptSession` exists because `refreshRepositoryWorkspace` lags the push response; that the `setTimeout(0)` hook idiom dodges a StrictMode/act() timing issue; that the conflict-detection heuristic produces false positives on real prose; that the dashboard inventory cache is observed stale in practice; that governance/reasoning/workflow tabs harbor render-time crashes beyond the one confirmed in DecisionLifecycleTab; that other top-level projections have latent drift like the confirmed DecisionContext case; and that DecisionSessionRecoveryService may already emit a recovery flag the UI ignores. None of these were exercised against a running system.

**Where the audit is strong vs thin.** *Strongest:* the contract-fidelity, mock-fidelity, shell-boundary, and React-state/component dimensions — these carry the most high-confidence findings, several independently verified, and converge on the same root cause from different angles. The static dependency-graph dimension is also high-confidence and uniformly positive. *Thinner:* `backend abstraction` and `frontend abstraction` lean on `medium`-confidence judgment calls (e.g. ABS-3's "single-impl interfaces may be intended future seams," EXPL-1's incomplete read of `DecisionSessionRecoveryService.cs`); the inventory-cache staleness (PROJ-6) and UIAUTH severity-tier findings are `medium` confidence and partly rest on inferred-but-unobserved runtime behavior. Some findings were not high-impact enough to warrant verification (`verification: null`) and should be treated at their stated confidence rather than as re-confirmed fact.

**Out of scope, restated.** This pass makes no measured claims about runtime behavior, performance, latency, memory, re-render counts, or actual blast-radius at runtime. Several findings note a plausible runtime consequence (e.g. all-tabs-mount amplifying re-renders, eager hooks multiplying backend calls on repo switch, the 29-call `Promise.all` collapsing a governance panel); every such consequence is labeled speculation and is explicitly deferred to the second, instrumented pass. Where this audit references `audit.md` (R1–R9), it does so as corroboration of static findings, not as runtime evidence gathered here.

### The worst violations

**Two verified, high-severity authority leaks anchor this report.**

The most architecturally dangerous is **PROJ-2 / SHELL-1's backend twin**: `DecisionProjectionService` — a type whose name implies a read-only derived view — has quietly become the authority that decides whether execution may launch. The verification confirmed literally that `ExecutionContextService.cs:74-92` loops over `decisionProjection.Conflicts`, adds a `validationError` per conflict (`:84-85`), and sets `LaunchBlocked = validationErrors.Count > 0` (`:210`). The conflicts themselves are computed by brittle natural-language matching: a hard-coded `DirectivePrefixes` table (`DecisionProjectionService.cs:19-39`, e.g. `("do not use ", false)`), `TechnologyTerms`/`WorkflowTerms` keyword lists (`:41-89`), and `ParseDirective` doing `ToLowerInvariant` + `StartsWith` + `Contains(prefix + subject)` substring matching (`:694-918`). The service even stamps `OriginatingAuthority = "DecisionProjectionService"` (`:232`), making the misplacement explicit. This is business-critical gating logic positioned as a transport/read concern, and it is only as reliable as a lowercase substring match. (Interpretation: the position as an authority is confirmed; the heuristic's actual accuracy on real decision prose is speculation — it was not exercised with data.)

The second verified high-severity leak is **SHELL-1 / CONTRACT-1**: the Rust shell is an *active, lossy projector* for the two repository-projection commands, not the passive proxy the architecture claims. Verification confirmed the backend emits `DecisionSessionSummary` (`RepositoryDashboardProjection.cs:36`, `RepositoryWorkspaceProjection.cs:44`), the UI type expects `decisionSessionSummary` (`types/repositories.ts:88, :107`), but the Rust mirror structs omit it (`main.rs:29-42`, `:222-238`) with no `#[serde(flatten)]` catch-all. Because `list_repositories` and `get_repository_workspace` deserialize-then-reserialize through these typed structs, serde silently drops the field on the round-trip. The UI was then forced to mask it with `?? EMPTY_DECISION_SESSION_SUMMARY` (`SelectedRepositorySummary.tsx:131-134`), pushing backend-owned domain defaulting into the presentation layer — an authority leak in the wrong direction (the shell subtracts domain data the backend authored).

**The UI-side falsifications are real but bounded.** The cleanest is the six-way duplicated tone classification (UIAUTH-1 / EXPL-2): whether "Stale" or "Deferred" is a warning vs. danger is a governance/workflow semantic, encoded six times via inconsistent substring tables that each silently degrade to `neutral` (these are confidence:high but unverified — `verification` is null, so treat as stated). Tone itself is legitimately a presentation concern the backend should not emit as a color; the defect is that the *same* policy is forked and drifting, so one backend severity string can render different colors per tab. The operational-context re-parse (UIAUTH-2) and the lifecycle state-set gating (UIAUTH-4) are the same root cause: domain rules duplicated in React where a backend-owned typed projection already exists.

**Note on what is well-designed:** the shell's error/boundary-violation passthrough (`main.rs:2689-2701`), the `~140` Value-passthrough commands, the data hooks that render backend verdicts verbatim, and `lib/status.ts`'s total enum→tone Record are the correct authority model and should be the template the violations converge toward — not changed.

**Structural summary across both contract investigators.** The two dimensions independently converge on one root cause: **there are three-to-four hand-maintained representations of every DTO** (C# record → Rust shell struct → TS type → devTauriMock) with **zero generation and zero contract test**. Verification confirmed a repo-wide search for `JsonPropertyName|JsonConverter|JsonIgnore|JsonPropertyOrder` across `**/*.cs` returns **0 matches** — the wire shape is wholly implicit (member name + `JsonSerializerDefaults.Web` camelCasing, `Program.cs:64-65`). The Rust shell compounds this with two contradictory bridge strategies in one file: ~30 typed mirror structs (`main.rs:19-329`) that can silently drop fields, interleaved with ~140 `serde_json::Value` passthrough commands that cannot drift by construction (CONTRACT-2, verified). The reasoning domain proves correct hand-maintenance is *possible* (11 enums and all records map 1:1), but the drift in decisions/decision-sessions/repositories/continuity proves it is not *reliable*. The one safety net that could catch TS-side divergence — `tsc -b` — was reported broken in this checkout (audit R9), so even compile-time checking is inert.

**Verdict on generation.** Contracts **should become GENERATED from a single source.** The evidence is decisive: every confirmed runtime defect in this audit (the `get_decision_context` white-screen, the `decisionSessionSummary` strip) is a *drift* defect, not a logic defect, and each is individually invisible because each layer's tests use that layer's own truth as the oracle. The C# DTOs are the natural canonical source (they own serialization). The recommended path is to emit OpenAPI/JSON Schema from the .NET minimal-API endpoints and generate the TS types (and the Rust structs, if kept) from it — and, per CONTRACT-2/STRUCT-4/SHELL-3, collapse the Rust mirror structs to `Value` passthrough so there are only **two** surfaces to reconcile, not four. The lower-risk interim is a CI contract test that boots the backend and diffs each endpoint's real JSON keys (recursively, not just top-level) against the declared TS types — promoting audit §2's drift scanner from throwaway to gate. Caveat (speculation): OpenAPI emission fidelity for these deeply nested polymorphic records is unverified and may require annotation work; turning it on will surface further latent drift, which is the point but will create churn. **Do not** continue hand-patching individual fields — that strategy is what created the drift.

## Mock Fidelity Report

**The dev Tauri mock encodes FRONTEND EXPECTATIONS (the TS types), not BACKEND REALITY.** This is the single largest blind spot in the test architecture, and it is proven definitionally, not inferred.

**Evidence (all verified).** The canonical proof is `get_decision_context` (MOCK-1, verified): the mock collapses both `get_decision_context` and `build_decision_context` to the *same* wrapped `DecisionContextSnapshot` shape (`devTauriMock.ts:4727-4729`, seeded by `createDecisionContext` returning a nested-context snapshot at `:416`). But the live GET returns the flat `DecisionContext` (`DecisionEndpoints.cs:72-79` → `BuildContextAsync` → `Task<DecisionContext>`, `IDecisionContextService.cs:7`); the snapshot wrapper is produced only by the POST/snapshot path (`DecisionContextService.cs:63-80`). Because the mock matches the (wrong) TS type rather than the GET endpoint, every dev/visual test passes while the real GET white-screens at `DecisionLifecycleTab.tsx:268`. The mock is not merely stale — it is *authored to satisfy the type*, and the type was never reconciled with what the backend emits. The same pattern holds for `decisionSessionSummary`: the mock includes it (`devTauriMock.ts:239, 266, 2804`) so dev rendering shows data the real Rust shell silently discards (CONTRACT-1, verified).

**Why CI cannot catch this (all verified).** Three structural facts compound:
1. **No contract test or schema-sharing exists** (MOCK-2, verified): a repo-wide grep for `openapi|swagger|AddOpenApi|MapOpenApi|NSwag|TypeGen|swashbuckle` returns **0 files**. The closest "endpoint test," `BackendEndpointDispositionTests.cs:10-44`, asserts only route presence/family/no-duplicates — it never deserializes a response body or compares it to any frontend type.
2. **The entire UI suite is driven exclusively by the mock** (MOCK-3, verified): `render.tsx:5-20` installs `installDevTauriMock()` for every component test, and `app.smoke.test.tsx` mounts the real App + real components against it — so the most integration-shaped tests in the repo still use a frontend-authored oracle. As the audit phrases it (and verification confirmed the literal text), the mock "validates the UI against itself."
3. **The mock returns pre-built objects via `structuredClone`** (`devTauriMock.ts:3699-3700`, verified, invoked 83×) and the production transport does **no runtime schema validation** (`tauri.ts:35-46`, the `<T>` is a compile-time cast only). So even if the mock's object *shape* were correct, the mock skips the JSON round-trip the real transport performs — making serialization-format drift (enum casing, `DateTimeOffset` vs ISO string, `Guid` casing, the `{value}` record-struct wrapping from CONTRACT-4) structurally invisible. The mock is thus *lower-fidelity than a recorded JSON fixture would be.*

**What is genuinely well-designed (leave alone):** the error/boundary-violation path *is* characterized at the real parse boundary — `tauri.ts:48-85` parses a stringified backend error payload with a runtime type guard, and `transport.test.ts:358-446` feeds a real `JSON.stringify`'d payload through the reject path. The certification fixture (`test/fixtures/certification.ts:9-41`) is small, typed, and exhaustive over a closed enum. These show the project already knows how to write a real boundary contract test; the pattern just needs extending to success-path response shapes.

**Recommended drift-prevention strategy (two complementary moves):**
1. **Make the mock a derived artifact, not a parallel source of truth.** Have the backend test run serialize each top-level projection DTO with the production `JsonSerializerOptions` and emit golden JSON fixtures; build the mock's response map from those fixtures (parsed through `JSON.parse` on return so it round-trips real serialized shapes), or add a CI test diffing the mock's emitted shapes against the goldens. This kills both the shape-drift and the `structuredClone` serialization blind spot.
2. **Add at least one real-transport test tier.** Promote audit §2's drift scanner to a committed contract/smoke test that exercises the real boundary, so a green suite cannot coexist with shape drift. Keep the existing fast mock-driven characterization tests for behavior/interaction — they are valuable — but stop treating them as contract coverage.

Sequence move 1 before re-sourcing the mock so the goldens exist first. **Caveats (from the findings):** fixtures must be captured from a known-good repository state (audit R8 noted a corrupt test repo) or they bake in bugs; a real-backend test tier adds CI infrastructure and startup-flakiness cost; and re-sourcing the mock will "break" tests that currently encode wrong shapes — that breakage is the signal you want.

## Refactor Roadmap

The findings cluster into eight evidence-supported programs. Each program is named, its member findings listed, and its through-line stated.

- **Program A — Projection normalization & authority re-grounding.** STRUCT-1, STRUCT-4, PROJ-1, PROJ-2, PROJ-3, PROJ-4, PROJ-5, PROJ-6. Through-line: stop "Projection" from naming three roles, relocate the execution-gating authority out of `DecisionProjectionService`, and make each read-model have one owner.
- **Program B — Contract single-source & generation.** CONTRACT-2/3/4/5 (decisions dim), CONTRACT-2/3/4/5 (exec dim), STRUCT-4, SHELL-3. Through-line: replace 3-4 hand-mirrored contract copies with one canonical source (the C# DTOs), generate TS, and collapse the Rust mirror tier to `Value` passthrough.
- **Program C — React state simplification.** STATE-1, STATE-2, STATE-3, STATE-4, STATE-5; supported by HOOK-1, HOOK-2, HOOK-3. Through-line: one owner per server slice, no `setData` escape hatch, derive instead of cache, one error model.
- **Program D — Component decomposition.** COMP-1, COMP-2, COMP-3, COMP-4, COMP-5 (the always-mounted/memo concern), STATE-6. Through-line: break the two god components into per-feature containers and conditionally mount tabs.
- **Program E — Resilience / error-isolation.** RES-1, RES-2, RES-3, RES-4, RES-5. Through-line: one backend error-shaping seam and one per-tab UI isolation seam; consistent empty-vs-error contracts; partial-failure tolerance.
- **Program F — Testing modernization.** TEST-1, TEST-2, TEST-3, TEST-4, MOCK-1, MOCK-2, MOCK-3. Through-line: introduce a cross-layer contract test, make the mock a derived artifact, and add a thin real-transport integration slice.
- **Program G — Dependency / abstraction reduction.** ABS-1, ABS-2, ABS-3, ABS-4, STRUCT-2, STRUCT-3, HOOK-1, HOOK-2, DESIGN-1, STRUCT-5/EXPL-layering. Through-line: honest DI contracts, one composition pattern, one resource/action hook factory, prune dead seams.
- **Program H — UI authority re-grounding (presentation vs domain).** UIAUTH-1, UIAUTH-2, UIAUTH-4, EXPL-1, EXPL-2, EXPL-3. Through-line: stop the React/adapter layers from re-deriving backend domain verdicts (tone tiers, markdown classification, lifecycle gates, materialization thresholds).

**The runtime/performance program is explicitly deferred to a measured second pass.** Several findings note render-frequency, eager-fetch multiplication, and blast-radius concerns (COMP-5, COMP-4, COMP-2 speculation), but every such note is flagged as unverified static reasoning. No re-render counts, fetch-amplification measurements, or latency numbers exist in this audit, so no performance refactor is sequenced here; it requires a profiling pass first.

### Sequenced milestones

Goal: fix `tsc -b` (audit R9) and add the cross-layer contract test (TEST-1) — golden JSON emitted from the production serializer, parsed in Vitest with `satisfies` + recursive key-diff — starting with `DecisionContextSnapshot`/`DecisionContext`, `RepositoryWorkspaceProjection`, and `DecisionSessionLifecycleProjection`. Reduces complexity: turns the highest-risk seam into an executable contract instead of two self-referential test suites. Preserves behavior: tests/fixtures only, no production code touched. Avoids churn: establishes the regression net that every later milestone leans on, so subsequent refactors are guarded rather than speculative.

**Milestone 1 — Stop the bleeding: the three confirmed crashes/data-loss (top of the inventory).**
Goal: land SHELL-1 (Value passthrough for the two repository projections), PROJ-3/MOCK-1 (align the GET decision-context contract and split the mock cases), and RES-2 (per-tab error boundaries). Reduces complexity: removes the lossy shell projector and the dual-shape decision-context type, and replaces whole-app white-screens with per-tab degradation. Preserves behavior: SHELL-1 strictly widens what passes through; RES-2 is additive; PROJ-3 keeps the snapshot/POST path intact and only corrects the live-read type and consumer access path. Avoids churn: these are small, surgical edits whose correctness is now pinned by Milestone 0's contract test (Milestone 0 would itself have failed on PROJ-3/SHELL-1).

**Milestone 2 — Backend error-shaping seam + empty-vs-error convention (Program E backend slice).**
Goal: RES-1 (one `IExceptionHandler` producing the `{error,boundaryViolation?}` envelope, delete per-endpoint inline try/catch), then RES-3 (one convention for "absent session"). Reduces complexity: removes ~7 divergent per-file catch sets and closes the IOException-as-500 leak class. Preserves behavior: behavior-unchanged for already-handled exceptions; strictly improves the unhandled ones. Avoids churn: a single owned seam replaces N hand-maintained ones, so future endpoints inherit the contract for free rather than re-deriving it.

**Milestone 3 — Contract single-source & shell convergence (Program B core).**
Goal: collapse the Rust mirror tier to `Value` passthrough (SHELL-3 / CONTRACT-2 exec dim), then stand up TS-from-C# generation (or, as an interim, promote the contract scanner to CI) and correct the known TS drifts (CONTRACT-3/4/5 decisions dim; CONTRACT-3/4 exec dim). Reduces complexity: deletes ~250 lines of mirror structs and removes a whole drift class; reduces four contract copies toward one generated artifact. Preserves behavior: the UI TS types already define the full shape, and `Value` passthrough is transparent through Tauri. Avoids churn: generating from the canonical C# source means the remaining per-field TS corrections become a one-time reconciliation, not recurring hand-patches.

**Milestone 4 — Mock fidelity & test tiers (Program F remainder).**
Goal: re-source `devTauriMock` from backend-derived JSON goldens (MOCK-3/TEST-2), add the transport degenerate-case tests (TEST-3), and a thin real-hook integration slice for 2-3 high-risk tabs (TEST-4). Reduces complexity: the mock stops being a parallel hand-maintained source of truth and becomes a derived artifact. Preserves behavior: behavioral characterization tests stay; only their data oracle changes. Avoids churn: sequenced after Milestone 3 so the goldens already exist; surfacing latent drift here is the intended signal, not rework.

**Milestone 5 — UI authority re-grounding (Program H).**
Goal: collapse the six tone classifiers into one resolver / backend-emitted tier (UIAUTH-1/EXPL-2), project the typed operational-context document (UIAUTH-2), drive lifecycle gates off `isAllowed` (UIAUTH-4), and stop adapters re-deriving backend outcomes (EXPL-1/EXPL-3). Reduces complexity: removes duplicated domain judgments straddling C#/TS and the silent 'neutral' fallthrough. Preserves behavior: characterization tests for explainability adapters and the decision panels pin current output, so changes are explicit and test-guarded. Avoids churn: leaves the genuinely passive renderers untouched and pushes logic toward the backend rather than rearranging the view layer.

**Milestone 6 — State ownership & hook consolidation (Program C + Program G hook slice).**
Goal: introduce the `useRepositoryResource` factory with the stale-response guard baked in (HOOK-1/HOOK-3) and `useAsyncAction` (HOOK-2); then unify execution state on one owner (STATE-1, STATE-4), remove the `setData` escape hatch (STATE-2), colocate the artifact draft (STATE-3), and pick one error model (STATE-5). Reduces complexity: removes ~1000+ lines of fetch boilerplate, three copies of the execution field list, and 11 funnel effects. Preserves behavior: keep each named hook's public surface; the factory reproduces the `setTimeout(0)` deferral and empty-state semantics; existing characterization tests validate. Avoids churn: done hook-by-hook/domain-by-domain (workspace first), so the public contract callers depend on never moves.

**Milestone 7 — Component decomposition (Program D).**
Goal: extract per-tab containers and conditionally mount tabs (STATE-6, COMP-2), decompose `App.tsx` into per-feature controllers (COMP-1, COMP-3), split `DecisionLifecycleTab` and lift its hooks into a shared controller (COMP-4), and apply `React.memo` at the tab seam only after prop shapes are fixed (COMP-5). Reduces complexity: dissolves the two god components and turns the 27-33-prop interfaces into cohesive controller objects. Preserves behavior: done incrementally one domain/tab per PR, behind the now-comprehensive test suite (Milestones 0/4); interdependent handlers move together. Avoids churn: deliberately last, because conditional mounting and controller extraction are easiest once state ownership (Milestone 6) and isolation seams (Milestone 1's boundaries) already exist — doing it earlier would mean reshaping the same files twice.

Reduces complexity: removes naming ambiguity, duplicated mappers, fragile build coupling, and unused seams. Preserves behavior: PROJ-2 must preserve identical conflict output (write characterization tests first); the rest are mechanical/internal. Avoids churn: grouped last because the rename (PROJ-1) and authority extraction (PROJ-2) are wide compile-surface changes best done once the contract and test foundations are stable; SHELL-4 (route-table abstraction) is explicitly **not** scheduled — its risk of over-engineering a working bridge exceeds the benefit absent observed route churn.

## Things Explicitly Not Worth Refactoring

These represent good design already in place. Touching them is churn for its own sake.

- **The backend project dependency graph.** A clean acyclic layered DAG on a genuine `Core` base (in-degrees: Core 8, Decisions/Continuity 5, etc.; no back-edges, no cycles). Core holds only foundation primitives, so its high fan-in is healthy. (`src/CommandCenter.*/*.csproj` adjacency.) Leave the layering alone.
- **The UI module graph and single transport seam.** Strict `features → hooks → api → tauri → types` flow with no presentation-imports-transport violations and no cycles; `api/tauri.ts` is the **only** file importing `@tauri-apps/api/core`, so swapping transport touches one file. (`src/CommandCenter.UI/src` import-edge counts; `api/tauri.ts`.)
- **The `AddX()` DI extension pattern and `Program.CreateApp` override hook.** Each domain owns its `Extensions/ServiceCollectionExtensions.cs`; `configureServices` runs after domain registration so tests substitute real alternates. This is the template the rest should converge to (it is the fix vehicle for ABS-1), not a thing to change. (`Program.cs:27-63`; per-domain extension files.)
- **The Tauri error/boundary-violation contract.** `response_error` re-serializes the structured `boundaryViolation` verbatim, and `api/tauri.ts:48-85` reconstructs a typed `BoundaryViolationProjection` with a runtime guard. The shell makes zero domain interpretation of failures. This is the reference pattern for RES-5/SHELL-1, not a refactor target. (`main.rs:2689-2701`, `:324-329`; `tauri.ts:48-85`.)
- **The `Value`-passthrough command tier in the shell.** ~140 commands forward opaquely and cannot drift by construction; they are the model SHELL-1/SHELL-3 should converge **toward**. (`main.rs:2645-2687`; workflow/continuity/decision-session command ranges.) Leave untouched.
- **Backend lifecycle/process supervision.** `start_backend`/`wait_for_backend`/`BackendProcess` Drop and `backend_executable_path` are genuine host-shell concerns with no domain leak. Explicitly out of scope for any "make the shell more passive" work. (`main.rs:2737-2786`, `:15-17`, `:342-346`.)
- **`useShellState` navigation authority.** Single, minimal source of truth for selection/tab/palette with per-repo memory; derived paths computed not stored; encapsulated intent-named callbacks, pinned by `shellState.test.tsx`. It is the template for the data-layer refactors (COMP-1, STATE-*), not a target. (`state/shellState.ts:45-178`.)
- **`useExecutionSession`'s request-sequencing.** Correct, well-scoped `useRef` concurrency control that drops stale writes and localizes race handling inside the hook. Its guard should be **generalized** into the HOOK-1 factory (HOOK-3), not removed. (`useExecutionSession.ts:14-15,:45-69`.)
- **The thin data hooks (`useDecisionLifecycleEligibility`, `useExecutionGitEligibility`, `useWorkflowHealth`, `useDecisionQuality`).** Pure fetch-and-store over backend projections with zero client domain computation — the correct authority boundary. (`hooks/use*Eligibility.ts`, `useWorkflowHealth.ts:41-53`.)
- **`lib/status.ts` status→presentation tables.** Total, compile-checked `Record<Enum, StatusPresentation>` maps off closed backend enums; a renamed member is a TS error. This is the template UIAUTH-1/EXPL-2 should rewrite the six tone-scanners toward. (`lib/status.ts:21-94`.)
- **The reasoning enum taxonomy and records.** All 11 C# enums map 1:1 to the TS unions value-for-value (via the global `JsonStringEnumConverter`), and the reasoning records map field-for-field with consistent nullability. Reasoning is the healthiest contract domain; churning it adds risk for no gain. (`ReasoningEnums.cs`; `Program.cs:65`; `reasoning.ts:1-453`.)
- **The design-system layer and explainability **renderers**.** `components/design` is cohesive and widely adopted (~60 import sites); the explainability renderers are genuinely passive (every tone read from a field, never computed). Push logic OUT of adapters toward the backend (Program H), never INTO these components. (`components/design/index.ts`; `HealthView.tsx:28`, `ConstraintViewer.tsx:28-30`, `CertificationFindingsView.tsx:38`.) The single normalized `Explanation` contract and its characterization tests are likewise worth keeping as the anchor for any adapter refactor.
- **The markdown artifact projectors' internals.** `DecisionArtifactProjectionService`/`ReasoningArtifactProjectionService` are pure renderers that explicitly disclaim authority ("not authoritative lifecycle state"). Only their **name** is a problem (PROJ-1); the internals are clean. (`ReasoningArtifactProjectionService.cs:71`.)
- **Interfaces with genuine alternates.** `IArtifactStore`, `IExecutionProvider`, `IGitService`, `IProcessRunner`, `IDecisionRepository`, and the minimal `IRepositoryService` all have real second implementations or in-use test doubles swapped at real sites. These are load-bearing seams; the dependency-reduction work (ABS-1/2/3) must **not** collapse them. (`MemoryArtifactStore`, `FakeExecutionProvider` at 8 registration sites; `IRepositoryService.cs:3-10`.)
- **The backend real-HTTP endpoint, disposition, and read-only/no-mutation tests.** 21 files exercise the full HTTP stack via `HttpClient`; `BackendEndpointDispositionTests` locks the route surface; `DecisionSessionEndpointTests` asserts byte-for-byte no-mutation on workflow GETs. These are strong existing seam coverage on the .NET side and should be preserved as-is. (`WorkflowEndpointTests.cs:46-91`; `BackendEndpointDispositionTests.cs:11-85`; `DecisionSessionEndpointTests.cs:19-103`.)
- **The behavioral authority-regression tests.** `workflowAuthority.test.ts` and `decisionTransparencyAuthority.test.ts` encode the "React/shell must not compute backend semantics" boundary as falsifiable source-scanning guards. They are the highest-value form of authority test here and underpin Program H's safety. Keep them.
