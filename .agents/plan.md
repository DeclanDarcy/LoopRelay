# Plan Authoring to Execution to Decision Loop Implementation Plan

## Purpose

Implement the proposed Plan Authoring -> Execution -> Decision Loop design as the next concrete Command Center evolution.

The product behavior is intentionally narrow: when a repository has no `.agents/plan.md`, the repository view opens a Plan Authoring screen. From that screen the user writes roadmap/spec inputs, asks Codex to write and revise a plan, executes the accepted plan, and then stays in a user-mediated decision loop while operational execution continues.

This plan replaces the broader repository knowledge and adaptive intelligence roadmap. Those ideas are future possibilities, not scope for this implementation.

## Target Lifecycle

1. A repository with no `.agents/plan.md` shows Plan Authoring instead of the normal dashboard.
2. The user writes a Roadmap and zero or more Specs, then clicks Write Plan.
3. An Operational Codex process writes `.agents/plan.md` and streams output to the UI.
4. The user can revise the plan repeatedly by submitting Feedback to the same held-open planning process.
5. Execute Plan closes the planning process, extracts milestones, commits and pushes, starts operational execution, and captures `.agents/handoff.md`.
6. A separate Decision Codex process starts from `.agents/operational_context.md`, receives the operational handoff, and streams proposed decisions.
7. The UI swaps the streamed decision output into an editable textarea. The user edits and submits.
8. Submission persists `decisions/decisions.000N.md`, continues operational execution with `ContinueExecution.Render(plan, handoff, decisions)`, captures the next handoff, and routes the next decision turn.
9. The router either reuses the warm Decision process or transfers operational context into a fresh Decision process before asking for next decisions.

## Governing Invariant

Operational Sessions and Decision Sessions are distinct session roles, but both are backed by real Codex processes through the shared `CommandCenter.Agents` runtime.

- Operational Sessions do code work, repository edits, Git, commits, pushes, handoffs, and operational evidence.
- Decision Sessions reason over operational handoffs and emit user-editable decisions. They do not code, commit, push, or run operational orchestration.
- `CommandCenter.DecisionSessions` must not reference Execution's operational orchestration such as `ExecutionSessionService` or Git lifecycle services.
- `CommandCenter.Execution` and `CommandCenter.DecisionSessions` both depend on the role-agnostic `CommandCenter.Agents` runtime for process spawning, streaming, and lifecycle.
- Session role selects sandbox, effort, prompt, permissions, and lifecycle policy. Agent Runtime receives rendered prompt text and metadata; it does not select or compose canonical prompt text.

## Build Order

The design depends on prerequisites that must land before the UI flow can be certified:

0. Ratify the invariant and prompt/runtime authority boundaries.
1. Extract `CommandCenter.Agents` as the role-agnostic process/session runtime with no product behavior change.
2. Add persistent or resumable Codex sessions that can accept multiple sequential stdin turns and expose per-turn completion.
3. Make Decision Sessions Codex-backed, with deterministic decision services retained only as offline/fallback behavior.
4. Wire the router to the active Decision process so `Continue` reuses the warm process and `Transfer` seeds a fresh process.
5. Add the Plan Authoring UI and repository-scoped orchestrator that compose the above pieces into the lifecycle.

## Prompt Authority

`CommandCenter.Core.Prompts` is the canonical prompt API. Authored prompt files live under `src/CommandCenter.Core/Prompts/*.prompt` and are generated by `Lib.Prompts` at build time. Runtime, UI, tests, compatibility adapters, Execution, DecisionSessions, Continuity, Workflow, and Backend must not duplicate canonical prompt strings.

Canonical prompt catalog and generated signatures:

- `WritePlanForNewCodebase.Text`
- `WritePlanAgainstCodebase.Text`
- `ExtractMilestones.Text`
- `ProduceOperationalDelta.Text`
- `UpdateOperationalContext.Text`
- `RevisePlan.Render(feedback)`
- `StartExecution.Render(plan)`
- `GetNextDecisions.Render(handoff)`
- `StartDecisionSession.Render(operationalContext)`
- `StartDecisionSessionFromTransfer.Render(operationalContext)`
- `ContinueExecution.Render(plan, handoff, decisions)`

Every agent turn records prompt name, generated type, `SourceHash`, session role, workflow phase, rendered input artifact identities, and produced artifact identities.

## Process Model

Operational profile:

- Codex can write the repository and `.agents` artifacts.
- Intended sandbox: workspace-write with approvals off.
- Used for planning, milestone extraction, execution, continuation, and operational context updates.

Decision profile:

- Codex has zero operational permissions.
- Intended sandbox: read-only, approvals never, no MCP/tools where supported.
- Used for decision-session start, next decisions, and operational-delta production.

Effort tiers:

- ExtraHigh for plan writing/revision, milestone extraction, operational context update, and decision-session start or transfer.
- Medium for operational execution and continuation.
- The exact `model_reasoning_effort` values must be validated against the installed Codex version.

Process lifetimes:

- One-shot turns are acceptable for milestone extraction, start execution, continue execution, and operational context update.
- Held-open persistent turns are required for plan authoring/revision and the Decision process reuse path.
- If persistence cannot be implemented, the governed fallback is one-shot planning revision seeded with current plan and feedback, and transfer-only Decision turns.

## Repository Orchestrator

Add a singleton, per-repository orchestrator in Backend or a new composition-root project. It owns live process handles and transient run state across HTTP calls:

- open planning process;
- open decision process;
- cached plan, current handoff, current decisions, iteration counter, and router inputs;
- planning, execution, and decision SSE channels;
- disposal on cancellation, error, repository deselection, and application shutdown.

The orchestrator composes `CommandCenter.Agents`, `IArtifactStore`, Git services, decision lifecycle policy, transfer eligibility, memory cache, and existing domain services. It does not become semantic authority for planning, execution, decisions, continuity, Git, contracts, or UI behavior.

Repository visibility gate:

- `GET /api/repositories/{id}/plan/status` returns `{ planExists, state }`.
- `planExists == false` drives Plan Authoring.
- Add `PlanAuthoring` and `ExecutingPlan` as repository-level lifecycle states or map them through an equivalent repository lifecycle projection.

## Artifact Protocol

All repository writes go through `IArtifactStore` or existing repository artifact abstractions.

- `.agents/specs/roadmap.md`: written from the Roadmap textarea.
- `.agents/specs/s{n}.md`: written from each Spec textarea.
- `.agents/plan.md`: written by Codex; orchestrator verifies existence.
- `.agents/operational_context.md`: copy of `plan.md` at Execute Plan; rewritten on transfer.
- `.agents/handoff.md`: written by each operational turn; verified and then moved into history.
- `.agents/handoffs/handoff.000N.md`: run-scoped handoff history.
- `.agents/decisions/decisions.000N.md`: persisted edited decision submissions.
- `.agents/operational_delta.md`: captured Decision-process output during transfer.

This flow intentionally bypasses the current `AwaitingAcceptance` handoff gate. The human gate moves to decision Submit.

## API And Streams

Add endpoints following the existing repository route and SSE conventions:

- `GET /api/repositories/{id}/plan/status`
- `POST /api/repositories/{id}/plan/write`
- `POST /api/repositories/{id}/plan/revise`
- `GET /api/repositories/{id}/plan/stream`
- `POST /api/repositories/{id}/plan/execute`
- `GET /api/repositories/{id}/execution/stream`
- `GET /api/repositories/{id}/decision/stream`
- `POST /api/repositories/{id}/decision/submit`

All streams use the existing `text/event-stream` shape with JSON `data:` payloads, turn boundaries, completion, failure, and reconnect/replay metadata where needed.

## UI

Add `src/CommandCenter.UI/src/features/planning/PlanAuthoringScreen.tsx` plus API and stream hooks.

Required controls and states:

- Roadmap textarea, required for Write Plan.
- Specs textareas, unbounded via Add Spec.
- New Codebase checkbox, default unchecked.
- Write Plan button, disabled until Roadmap has non-empty text.
- Read-only planning stream and rendered plan view.
- Copy icon button for the rendered plan.
- Feedback textarea and Revise Plan button, enabled only when Feedback is non-empty.
- Execute Plan button.
- Decision stream that becomes an editable decisions textarea on turn completion.
- Submit button for edited decisions.

UI state machine:

`Authoring -> Planning -> PlanReady -> Revising -> PlanReady -> Executing -> DecisionStreaming -> DecisionEditable -> Submit -> ContinueStreaming -> DecisionStreaming`

The same screen hosts the lifecycle; do not split the user into unrelated planning, execution, and decision pages for this flow.

## Definition Of Done

From a repository with no `.agents/plan.md`, the user can author roadmap/spec inputs, generate and revise a plan through a held-open Operational Codex process, execute the plan, extract milestones, commit and push, start operational execution, and then sustain a decision loop where a separate zero-permission Decision Codex process proposes user-editable decisions. The router measurably reuses the warm Decision process or transfers operational context into a fresh Decision process based on active-session token pressure. All prompts come from `CommandCenter.Core.Prompts` generated by `Lib.Prompts`, and Execution and DecisionSessions reach Codex only through `CommandCenter.Agents`.
