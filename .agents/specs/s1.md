# Milestone 1 Deep Dive — Canonical Contracts and Vocabulary

# Purpose

Milestone 1 establishes the canonical language and contracts of the orchestration system.

Nothing in this milestone changes workflow behavior.

Nothing in this milestone changes orchestration behavior.

Nothing in this milestone migrates existing workflows.

Instead, this milestone establishes the architectural vocabulary that every subsequent milestone will implement.

At the completion of Milestone 1, every existing workflow should be describable using the same canonical concepts without yet changing how those workflows execute.

---

# Objectives

Create explicit architectural contracts for:

* Workflow
* Workflow Chain
* Workflow Identity
* Stage
* Transition
* Transition Dependency
* Input Gate
* Output Gate
* Workflow Entry Gate
* Workflow Exit Gate
* Product
* Product Identity
* Product Ownership
* Product Authority
* Transition Result
* Effect
* Blocker
* Recovery
* Workflow State
* Stage State

These become the authoritative architectural vocabulary for the remainder of the program.

---

# Non-Goals

Milestone 1 does **not**:

* migrate workflows
* execute prompts
* change CLI behavior
* change persistence
* implement orchestration
* implement workflow chaining
* simplify Roadmap CLI
* refactor Plan
* refactor Execution

The repository behavior remains identical.

---

# Guiding Principles

## Explicit over implicit

Every architectural concept should have one explicit meaning.

---

## Semantic over implementation

Contracts describe:

what something **is**

not

how it is currently implemented.

---

## Repository-owned

Every runtime concept exists within the repository.

Nothing depends upon hidden process state.

---

## Workflow-agnostic

Contracts must describe:

Traditional Roadmap

Eval Roadmap

Plan

Execution

equally well.

No workflow receives special treatment.

---

Milestone 1 produces:

# Success Criteria

Milestone 1 is complete when:

* every architectural concept has one authoritative definition
* workflows are described entirely through canonical contracts
* stages are defined semantically
* transitions are prompt-driven contracts
* products are separated from artifacts
* gates are unified
* dependencies are explicit
* effects are separated from transitions
* outcomes use one shared vocabulary
* state is defined independently of persistence
* no production behavior has changed
* later milestones can implement orchestration without redefining architectural terminology

At completion, the project possesses a stable architectural language that every implementation milestone can rely upon without introducing new foundational concepts.
