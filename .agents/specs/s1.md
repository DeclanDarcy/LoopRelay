# Milestone 1 Deep Dive - Changed File Classification

## Purpose

Milestone 1 introduces the first executable stage of the post-execution review loop.

It deterministically classifies changed files from `git diff` so LoopRelay can identify likely non-implementation candidates without sending every file to semantic analysis.

The classifier is false-positive tolerant, but it must reliably route implementation code and machine-required files away from non-implementation review.

---

## Objective

Given a post-execution diff, produce a deterministic candidate list of changed files that may be non-implementation artifacts.

This milestone answers:

> Should this changed file be sent to semantic candidate confirmation?

It does not answer whether the file should be kept, deleted, synthesized, or reviewed by the HITL.

---

## Responsibilities

Milestone 1 owns:

* changed file discovery from `git diff`
* parallel deterministic classification of each changed file
* reliable exclusion of implementation code
* reliable exclusion of machine-required files
* recognition of sanctioned LoopRelay operational artifacts that should not enter review
* routing of likely or uncertain non-implementation files as candidates
* classification evidence for downstream explanation

---

## Boundaries

Milestone 1 does not own:

* semantic reasoning
* final confirmation that a candidate is non-implementation
* false-positive recording
* ledger persistence
* insight extraction
* HITL review
* deletion or retention decisions

The classifier is a router. It is not a preservation authority or repository acceptance gate.

---

## Implementation Intent

Classification should be deterministic and reproducible for the same repository state, diff, and policy setting.

The MVP is file-list oriented because the HITL described the flow as `git diff -> file list -> classify each file in parallel`. Broader mutation abstractions may be possible later, but they should not obscure the MVP shape.

The policy should favor recall for non-implementation candidates while maintaining the exclusion invariant for code and machine-required artifacts.

---

## Classification Outcomes

The deterministic stage should produce routing outcomes such as:

* excluded implementation artifact
* excluded machine-required artifact
* excluded sanctioned operational artifact
* non-implementation candidate
* uncertain candidate

Uncertain candidates proceed to semantic confirmation.

---

## Acceptance Criteria

Milestone 1 is complete when:

* LoopRelay can obtain the changed file list from the post-execution diff
* each changed file can be classified in parallel
* code files reliably avoid non-implementation review
* machine-required files reliably avoid non-implementation review
* likely or uncertain non-implementation files are routed for semantic confirmation
* classification output includes enough evidence to explain the route

---

## Dependencies

Depends on Milestone 0 for:

* artifact vocabulary
* policy setting location
* review-loop ownership

---

## Failure Modes

Avoid:

* extension-only classification when repository context is available
* semantic reasoning inside the deterministic stage
* treating candidates as confirmed non-implementation files
* allowing code or machine-required files to routinely enter HITL review
* broad repository mutation acceptance design
