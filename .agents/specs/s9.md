# Milestone 9 Deep Dive — Unified CLI, Legacy Retirement, and Architecture Certification

# Purpose

Milestone 9 completes the Canonical Orchestration Architecture program.

Every workflow has already been migrated.

The runtime already exists.

Repository resolution already exists.

Workflow chaining already exists.

This milestone makes the new architecture the authoritative public execution model.

It replaces the existing orchestration surfaces with one unified CLI, retires obsolete orchestration implementations, certifies behavioral equivalence, and establishes the Canonical Orchestration Architecture as the sole orchestration authority.

---

# Objectives

Deliver:

* one unified CLI
* one orchestration authority
* one workflow runtime
* one repository resolution engine
* one workflow controller
* one completion authority

while retiring:

* duplicate orchestration
* duplicate workflow progression
* duplicate completion ownership
* legacy execution-preparation orchestration
* obsolete CLI orchestration paths

without changing intended workflow behavior.

---

# Non-Goals

Milestone 9 does **not**:

* redesign workflows
* redesign prompts
* redesign products
* redesign storage
* redesign persistence
* redesign recovery
* redesign completion semantics
* implement execution loop-back
* implement concurrency
* implement confidence
* implement telemetry-driven orchestration

Architecture becomes authoritative.

Behavior remains equivalent.

---

# Guiding Principles

## One public entry point

The user should think in terms of:

```text
looprelay
```

not individual workflow executables.

---

## One orchestration authority

At the completion of this milestone there must be exactly one authority responsible for:

* workflow selection
* stage selection
* transition execution
* workflow progression
* workflow chaining
* orchestration recovery

No legacy orchestration path may remain authoritative.

---

## Compatibility without architectural duplication

Backward compatibility is acceptable.

Duplicate orchestration logic is not.

Compatibility layers should delegate.

They should never compete.

---

## Explainability remains mandatory

Every orchestration decision remains explainable.

The unified CLI must expose:

* workflow
* stage
* blockers
* authority
* evidence
* repository classification

without exposing implementation details.

# Architecture Completion Criteria

The Canonical Orchestration Architecture program is complete when:

* exactly one public CLI exists
* exactly one orchestration authority exists
* exactly one repository resolution engine exists
* exactly one workflow controller exists
* exactly one transition runtime exists
* exactly one completion authority exists
* all four workflows execute through identical orchestration
* two workflow chains are supported
* default invocation automatically selects the correct roadmap workflow
* automatic storage verification precedes orchestration
* workflow chaining is deterministic
* bounded workflow execution is preserved
* every orchestration decision is explainable
* every workflow can resume correctly
* every workflow uses semantic products instead of artifact coupling
* legacy orchestration has been retired
* compatibility remains intact
* behavioral parity has been certified
* architectural consistency has been certified
* the public orchestration contract has been frozen

At completion, LoopRelay no longer consists of independently evolved orchestration implementations. It becomes a single, coherent orchestration platform where every workflow is expressed declaratively, every transition executes through the same runtime, every workflow boundary is governed by the same gate model, and the repository itself becomes the authoritative source of orchestration state.
