# Milestone 6 Deep Dive - Planning Integration and Implementation-First Guidance

## Purpose

Milestone 6 pairs the review capability with conditional prompt guidance that prevents avoidable non-implementation artifacts before execution begins.

The goal is not to ban documentation. The goal is to ensure LoopRelay does not autonomously prescribe non-implementation files, documentation milestones, or theory-protection artifacts unless the HITL specifically requested them.

---

## Objective

Conditionally inject implementation-first guidance into planning and execution prompts through the explicit opt-in setting established in Milestone 0.

This milestone answers:

> How should LoopRelay steer agents away from autonomous non-implementation file generation while preserving explicit HITL intent?

---

## Responsibilities

Milestone 6 owns:

* connecting the explicit opt-in setting for non-implementation file generation to prompt policy composition
* injecting guidance into roadmap, milestone, plan, execution, and decision prompts where relevant
* discouraging autonomous non-implementation documentation
* discouraging documentation-centric milestone framing
* discouraging Architecture Tests, Golden Tests, and theory-protection artifacts not backed by executable evaluation
* preserving the exception for HITL-requested non-implementation documentation

---

## Boundaries

Milestone 6 does not own:

* file classification
* semantic confirmation
* ledger persistence
* HITL review workflow
* repository mutation acceptance
* broad enforcement outside prompt guidance
* banning documentation requested by the HITL

Planning guidance reduces avoidable proliferation. The post-execution review loop remains responsible for detecting what actually changed.

---

## Implementation Intent

Guidance should derive from a centralized policy composition mechanism rather than duplicated prompt text.

When the explicit opt-in is absent, prompts should hard-steer away from:

* Freeze Docs milestones
* Certification Docs milestones
* Governance Docs milestones
* Authority Docs milestones
* epic details that prescribe autonomous non-implementation files
* Architecture Tests
* Golden Tests
* other artifacts whose purpose is to protect theory rather than validate production-ready implementation through executable evaluation

The setting defaults to implementation-first behavior: unless the user explicitly opts in to non-implementation file generation, prompts should hard-steer away from autonomous non-implementation artifacts. The guidance must also state the positive rule: non-implementation documentation is allowed if and only if the HITL specifically requested it or the requested deliverable itself is non-implementation documentation.

---

## Prompt Targets

Apply guidance where it can affect artifact generation:

* roadmap generation
* milestone specification generation
* implementation planning
* execution prompts
* decision prompts that choose deliverables
* completion prompts where file review is summarized

---

## Acceptance Criteria

Milestone 6 is complete when:

* the user setting expresses explicit opt-in to non-implementation file generation and defaults to implementation-first guidance
* relevant prompts receive guidance from centralized policy composition
* planners avoid autonomous documentation-centric milestones when the explicit opt-in is absent
* execution guidance discourages non-implementation files not explicitly requested by the HITL
* Architecture Tests, Golden Tests, and theory-protection artifacts are discouraged unless backed by executable evaluation or explicitly requested
* HITL-requested documentation remains allowed

---

## Dependencies

Depends on Milestone 0 for:

* user setting architecture
* prompt policy composition
* shared vocabulary

This milestone can be implemented after the review loop foundation exists. It does not depend on repository mutation acceptance.

---

## Failure Modes

Avoid:

* duplicating policy text across prompts
* treating implementation-first mode as a documentation ban
* preventing explicitly requested documentation
* steering toward theory-protection tests that do not validate executable behavior
* making planners responsible for post-execution detection
