# Milestone 6 Deep Dive — Plan Workflow Migration

# Purpose

Milestone 6 migrates the existing Plan workflow onto the Canonical Transition Runtime and Workflow Orchestration Architecture.

Unlike Traditional Roadmap, which is primarily concerned with strategic preparation, Plan is the architectural bridge between roadmap formation and implementation execution.

Its responsibility is to transform a validated implementation-ready roadmap product into an execution-ready operational product.

The migration must preserve:

* implementation-first planning
* adversarial review
* warm-session refinement
* scoped artifact operations
* output validation
* publication semantics

while eliminating the pipeline-oriented orchestration that currently exists.

---

# Objectives

Migrate Plan so that:

* it becomes a first-class workflow;
* every stage executes through the Canonical Transition Runtime;
* workflow state replaces implicit pipeline progress;
* every transition is declarative;
* runtime owns orchestration;
* Plan satisfies one explicit downstream Execute contract.

---

# Non-Goals

Milestone 6 does **not**:

* migrate Execute
* redesign planning prompts
* redesign implementation-first guidance
* redesign adversarial review
* redesign milestone decomposition
* redesign operational context semantics
* redesign storage
* redesign completion

Only orchestration changes.

# Execute Entry Contract Certification

One of the most important deliverables of this milestone is the canonical Execute entry contract.

The Plan workflow must produce exactly one validated execution product set that downstream execution consumes regardless of which roadmap workflow produced it.

That contract must include, at minimum:

* Executable Plan
* Operational Context
* Execution Details
* Execution Milestone Set
* Execution Readiness

Each product must:

* be validated;
* be authoritative;
* belong to the current repository;
* belong to the current workflow chain;
* satisfy freshness requirements;
* satisfy ownership requirements.

Execute must never infer readiness from artifact existence alone.

It consumes the validated product contract.

---

# Partial Workflow Semantics

Unlike the existing pipeline, the migrated workflow must explicitly distinguish:

* never started
* planning in progress
* plan authored
* validation in progress
* validation complete
* execution preparation in progress
* partially materialized execution products
* execution-ready
* blocked
* cancelled
* failed
* completed

This eliminates the current ambiguity where existing outputs merely block a fresh run.

---

# Success Criteria

Milestone 6 is complete when:

* Plan executes entirely through the Canonical Transition Runtime
* the pipeline has become a true workflow
* stages represent domain progression
* transitions are declarative
* runtime owns orchestration
* warm sessions are runtime execution posture
* scoped artifact operations become ordinary transitions
* workflow state replaces implicit pipeline progress
* execution products become semantic products
* Execute receives one canonical entry contract
* partial Plan progress is explicitly representable
* Plan fully participates in repository resolution
* Plan fully participates in workflow chaining
* legacy pipeline orchestration is retired
* implementation-first planning behavior is preserved

At completion, Plan becomes the canonical architectural bridge between roadmap formation and execution. It no longer exists as a special-purpose pipeline, but as a first-class workflow expressed through the same orchestration architecture as every other workflow.
