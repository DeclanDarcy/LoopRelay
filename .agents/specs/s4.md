# Milestone 4 Deep Dive - Free-Form Insight Synthesis

## Purpose

Milestone 4 adds optional insight extraction for confirmed non-implementation files before HITL review.

The HITL should be able to read a compact synthesis before deciding whether to delete or keep non-implementation files. The MVP keeps this synthesis free-form so LoopRelay does not prematurely assume the optimal long-term structure.

---

## Objective

Generate compact, free-form synthesis from confirmed non-implementation files when those files contain meaningful information worth surfacing to the HITL.

This milestone answers:

> What useful information should the HITL see before deciding what to keep or delete?

---

## Responsibilities

Milestone 4 owns:

* selecting confirmed non-implementation files from ledger state for synthesis consideration
* generating concise free-form synthesis
* preserving enough source references for the HITL to understand where the synthesis came from
* making the synthesis available to the epic completion review
* avoiding structured knowledge assumptions in the MVP

---

## Boundaries

Milestone 4 does not own:

* candidate classification
* semantic candidate confirmation
* ledger disposition recording
* HITL keep/delete decisions
* promotion into a structured knowledge store
* schema design for future knowledge extraction
* repository acceptance

The synthesis supports review. It does not authorize repository changes.

---

## Implementation Intent

Insight extraction should be compact and useful, not comprehensive.

The synthesis should focus on meaningful implementation-relevant information that a HITL may want to retain after removing non-implementation files. It should avoid summarizing content that has no implementation value.

The MVP output remains free-form. If recurring structure becomes obvious through use, structured synthesis can be considered later outside this roadmap.

---

## Synthesis Inputs

Synthesis may consume:

* confirmed non-implementation ledger entries
* semantic confirmation rationale
* candidate file content
* relevant repository context

False positives should not be synthesized as non-implementation insight.

---

## Acceptance Criteria

Milestone 4 is complete when:

* confirmed non-implementation files can be considered for free-form synthesis
* useful synthesis is available before HITL review
* the synthesis remains compact and readable
* the synthesis does not require a structured schema
* source linkage is sufficient for HITL trust and follow-up
* false positives are excluded from synthesis

---

## Dependencies

Depends on Milestone 3 for:

* confirmed non-implementation ledger entries
* false-positive separation
* candidate provenance

Depends on Milestone 2 for:

* semantic rationale and uncertainty

---

## Failure Modes

Avoid:

* creating a structured knowledge system in the MVP
* summarizing false positives as if they were non-implementation files
* generating long documentation from non-implementation documents
* treating synthesis as a replacement for HITL authority
* promoting synthesis automatically into durable project specs
