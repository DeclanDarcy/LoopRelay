# Milestone 4 Deep Dive — Workflow Chaining and Unified Gates

# Purpose

Milestone 4 elevates the Canonical Transition Runtime and Repository Resolution Engine into a true orchestration engine.

Milestone 2 answered:

> **How does a transition execute?**

Milestone 3 answered:

> **What transition is eligible?**

Milestone 4 answers:

> **How do workflows compose into one deterministic execution chain?**

This milestone introduces no workflow migration.

Instead, it establishes the orchestration model that every migrated workflow will later use.

At completion:

* workflow boundaries become first-class runtime concepts;
* workflow chaining behaves identically to transition progression;
* downstream workflow execution becomes deterministic;
* repository progression becomes continuous.

---

# Objectives

Create one canonical workflow orchestration model responsible for:

* workflow entry
* workflow completion
* workflow chaining
* workflow boundaries
* downstream eligibility
* bounded execution
* automatic progression
* orchestration lifecycle

while preserving:

* explicit workflow identity
* repository ownership
* deterministic behavior
* explainability
* automatic storage verification
* future compatibility with dependency-driven concurrency

---

# Non-Goals

Milestone 4 does **not**:

* migrate workflows
* redesign prompts
* redesign persistence
* redesign recovery
* redesign storage
* implement concurrency
* implement execution-to-roadmap looping
* introduce confidence-based orchestration

The runtime still executes serially.

# Architectural Principles

## Workflow boundaries equal transition boundaries

The runtime must never distinguish between:

```text
Transition A
      ↓
Transition B
```

and

```text
Workflow A
      ↓
Workflow B
```

Both are governed by:

* output gates
* validated products
* input gates
* deterministic eligibility

The only difference is scope.

---

## Chaining is declarative

Workflows declare:

* downstream workflows
* required products

The runtime performs chaining.

Individual workflows never invoke downstream workflows.

---

## Workflow completion is a gate

A workflow completes only after:

* every required stage completes
* workflow exit gate succeeds
* required products validate
* downstream contract is satisfied

Completion is never inferred.

---

## Repository progression is continuous

From the user's perspective:

```text
looprelay
```

executes one continuous orchestration.

Internally:

the runtime advances through explicit workflow boundaries.

# Success Criteria

Milestone 4 is complete when:

* workflow chains are explicit
* workflow boundaries are first-class runtime concepts
* entry gates are unified
* exit gates are unified
* semantic products move between workflows
* workflow chaining is deterministic
* default invocation auto-chains
* `--eval` and `--traditional` force roadmap selection while preserving chaining
* single-workflow commands remain bounded
* workflow completion enables downstream eligibility instead of directly invoking downstream execution
* workflow controller is separated from transition runtime
* orchestration decisions are fully explainable
* future dependency-based concurrency is not architecturally prohibited
* execution-to-roadmap looping remains intentionally absent

At completion, the orchestration engine understands **how workflows compose**, while remaining completely independent of the internal implementation of any individual workflow. The remaining milestones become migrations of existing workflows onto this now-stable orchestration foundation.
