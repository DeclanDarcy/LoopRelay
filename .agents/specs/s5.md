# Milestone 5 Deep Dive — Traditional Roadmap Workflow Migration

# Purpose

Milestone 5 migrates the existing Traditional Roadmap workflow onto the Canonical Transition Runtime and Workflow Orchestration Architecture.

This is the first workflow migration.

It establishes the reference implementation that all later workflow migrations should emulate.

The objective is **not** to recreate the current Roadmap CLI.

The objective is to preserve its proven domain behavior while expressing it through the new orchestration architecture.

The existing Roadmap CLI is evidence.

The Canonical Orchestration Architecture is authority.

---

# Objectives

Migrate the Traditional Roadmap workflow so that:

* every stage executes through the Canonical Transition Runtime;
* every transition is declarative;
* workflow progression is managed exclusively by the Workflow Controller;
* repository products replace implicit artifact coupling;
* workflow state replaces incidental orchestration state;
* recovery, persistence, and explainability are inherited from the runtime rather than reimplemented.

---

# Non-Goals

Milestone 5 does **not**:

* migrate Plan
* migrate Execute
* implement Eval Roadmap
* redesign roadmap prompts
* redesign roadmap strategy
* redesign roadmap semantics
* implement concurrency
* redesign storage

Only the orchestration architecture changes.

The domain behavior should remain equivalent.

---

# Guiding Principles

## Preserve domain behavior

The architecture changes.

The roadmap decisions do not.

---

## Eliminate orchestration duplication

Roadmap should no longer own:

* prompt execution
* persistence sequencing
* transition bookkeeping
* output validation sequencing
* evidence sequencing
* workflow progression

Those become runtime responsibilities.

---

## Preserve rigor

Everything valuable in the current implementation remains:

* projection freshness
* prompt contracts
* input snapshots
* lifecycle validation
* artifact promotion
* blocker semantics
* recovery intent
* transition evidence

The rigor remains.

The implementation becomes smaller.

---

## Domain-first stages

Stages represent domain progress.

Not implementation mechanics.

Stages must never exist simply because:

* prompts are rendered
* state is persisted
* outputs are published

Those are runtime concerns.

---

# Success Criteria

Milestone 5 is complete when:

* Traditional Roadmap executes entirely through the Canonical Transition Runtime
* Workflow Controller owns workflow progression
* Repository Resolver owns workflow and stage discovery
* Workflow state replaces incidental orchestration state
* Transitions become declarative
* Prompt rendering, execution, persistence, evidence, and recovery are entirely runtime-owned
* Stages represent domain progression rather than implementation mechanics
* Products replace artifact coupling
* Workflow exit satisfies the canonical Plan entry contract
* Existing roadmap behavior is preserved
* Legacy orchestration is retired
* Traditional Roadmap becomes the architectural reference implementation for all remaining workflow migrations

At completion, the project possesses its first fully migrated workflow. Every remaining workflow migration (Plan, Execute, EvalRoadmap) should become substantially smaller because the orchestration infrastructure is already complete and proven.
