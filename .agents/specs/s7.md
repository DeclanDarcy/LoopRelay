# Milestone 7 Deep Dive — Execute Workflow Migration

# Purpose

Milestone 7 migrates the existing implementation execution loop onto the Canonical Transition Runtime and Workflow Orchestration Architecture.

This milestone replaces the current execution loop orchestration while preserving its operational behavior.

Unlike previous workflows, Execute is iterative.

The architecture must preserve iteration while eliminating bespoke orchestration.

Execution becomes:

* a canonical workflow;
* composed of domain stages;
* containing declarative prompt-driven transitions;
* executed entirely through the Canonical Transition Runtime.

---

# Objectives

Migrate Execute so that:

* execution becomes a first-class workflow;
* iteration is modeled explicitly rather than embedded inside orchestration code;
* every transition executes through the Canonical Transition Runtime;
* workflow state replaces implicit execution state;
* completion becomes one canonical workflow outcome;
* workflow behavior remains equivalent.

---

# Non-Goals

Milestone 7 does **not**:

* redesign implementation prompts
* redesign decision prompts
* redesign milestone philosophy
* redesign completion certification
* redesign Git behavior
* redesign handoff philosophy
* redesign operational context philosophy
* redesign completion review

Only orchestration changes.

---

# Guiding Principles

## Execute remains iterative

Execution is not a linear workflow.

Execution repeatedly advances implementation until completion.

The architecture must model iteration explicitly rather than hiding it inside orchestration logic.

---

## Decisions remain domain behavior

Decision generation remains part of execution.

Decision sessions are no longer orchestration.

They become one execution posture supported by the runtime.

---

## Completion remains rigorous

Milestone completion is not workflow completion.

Workflow completion still requires:

* completion review
* completion certification
* closure validation
* workflow exit gate

---

## Repository remains authoritative

Execution determines progress from repository state.

Never from hidden runtime memory.

# Canonical Iteration Semantics

Execution no longer relies on an implicit loop.

Instead, iteration becomes an explicit workflow behavior.

Every iteration produces one of two outcomes:

```text
Continue Iteration
```

or

```text
Attempt Workflow Completion
```

Workflow completion then determines:

```text
Continue

Blocked

Waiting

Cancelled

Failed

Certified Complete
```

This removes hidden control flow while preserving operational semantics.

---

# Repository Progress Semantics

Repository progress is evaluated explicitly.

The workflow must preserve the existing rigor around:

* real implementation changes;
* milestone advancement;
* decision progression;
* publication;
* completion evidence.

The workflow must never infer progress solely from prompt completion.

Progress remains repository-derived.

---

# Completion Authority

One architectural objective of this milestone is eliminating duplicate completion ownership.

The migrated Execute workflow becomes the single producer of:

* completion claim;
* completion review;
* completion certification;
* completion route;
* certified completion product.

Downstream consumers (including future roadmap looping) consume that product.

They do not recreate completion orchestration.

Milestone 7 is complete when:

* Execute executes entirely through the Canonical Transition Runtime
* execution becomes a true workflow
* iteration is explicit
* stages represent execution semantics
* transitions are declarative
* runtime owns orchestration
* decision sessions become execution posture
* products replace artifact coupling
* repository progress remains authoritative
* completion becomes one explicit stage
* one completion authority exists
* workflow state replaces implicit execution state
* Execute fully participates in repository resolution
* Execute fully participates in workflow chaining
* legacy execution orchestration is retired
* existing implementation behavior is preserved

At completion, all three existing production workflows—TraditionalRoadmap, Plan, and Execute—share one orchestration architecture. The remaining major implementation milestone is introducing EvalRoadmap as a new workflow that plugs into the same foundation rather than creating another orchestration model.
