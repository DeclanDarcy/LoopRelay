# Milestone 0 Deep Dive - Architectural Foundation

## Purpose

Milestone 0 prepares LoopRelay for the non-implementation file review capability without implementing the capability itself.

The milestone exists to remove ambiguity around vocabulary, settings, prompt policy flow, and ownership so later implementation work can stay focused.

It is not a repository governance milestone, a preservation platform milestone, or a general refactoring milestone.

---

## Objective

Establish the smallest architectural foundation needed for:

```text
git diff
-> changed file classification
-> semantic candidate confirmation
-> review ledger
-> optional free-form synthesis
-> HITL epic completion review
```

---

## Responsibilities

Milestone 0 owns:

* defining shared terminology for the review loop
* identifying the user setting for explicit opt-in to non-implementation file generation
* establishing the prompt policy composition point used by later prompts
* assigning ownership for post-execution non-implementation file review
* resolving architectural questions that would block implementation

The shared terminology should distinguish:

* implementation artifacts
* machine-required artifacts
* sanctioned LoopRelay operational artifacts
* non-implementation candidates
* confirmed non-implementation files
* false positives
* uncertain candidates
* HITL-requested non-implementation files
* free-form insight synthesis

---

## Boundaries

Milestone 0 does not implement:

* changed-file discovery
* deterministic classification
* semantic confirmation
* ledger persistence
* insight synthesis
* HITL review workflow
* prompt enforcement
* repository mutation acceptance
* commit or publication gating

Architectural work belongs in this milestone only when it directly reduces ambiguity for the requested review loop.

---

## Implementation Intent

Use existing LoopRelay architecture where possible.

The foundation should make repository policy and prompt guidance explicit without creating a broad repository governance framework. The policy setting should default to implementation-first behavior: LoopRelay hard-steers away from autonomous non-implementation file generation unless the user explicitly opts in. The setting must also preserve the rule that non-implementation documentation is allowed when the HITL specifically requested that documentation or deliverable.

Prompt composition should provide one place to inject policy guidance later, avoiding duplicated prompt language across planners and execution agents.

---

## Acceptance Criteria

Milestone 0 is complete when:

* the review-loop vocabulary is documented and used consistently
* the explicit opt-in setting for non-implementation file generation has a clear architectural home, including its default implementation-first behavior and HITL-requested documentation exception
* prompt policy composition has a clear architectural home
* ownership for the post-execution review loop is explicit
* later milestones can proceed without first deciding what terms, settings, or owners mean

---

## Dependencies

This milestone has no implementation dependencies.

It depends only on the recovered HITL intent and the existing LoopRelay architecture.

---

## Failure Modes

Avoid:

* broad repository governance design
* speculative policy families
* commit or publication gating
* future structured knowledge architecture
* unrelated cleanup
* terminology that obscures the file-review loop
