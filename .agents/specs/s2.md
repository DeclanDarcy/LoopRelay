# Milestone 2 Deep Dive - Semantic Candidate Confirmation

## Purpose

Milestone 2 confirms whether deterministic candidates are actually non-implementation files.

The HITL intent was not a broad evidence system. The MVP needs an execution agent to inspect each candidate and return structured confirmation.

---

## Objective

For every candidate routed by deterministic classification, produce a structured semantic disposition:

* confirmed non-implementation
* false positive
* uncertain

This milestone answers:

> Is this candidate in fact a non-implementation file?

It does not decide whether the file should remain in the repository.

---

## Responsibilities

Milestone 2 owns:

* invoking an execution agent for candidate confirmation
* requiring structured output
* distinguishing confirmed non-implementation files from false positives
* preserving uncertainty when the agent cannot decide
* producing concise rationale and evidence for ledger and review use

The structured output should include the candidate identity, disposition, rationale, and any uncertainty needed by downstream stages.

---

## Boundaries

Milestone 2 does not own:

* deterministic candidate routing
* ledger persistence
* false-positive reuse
* insight synthesis
* HITL keep/delete decisions
* prompt guidance
* repository acceptance

Semantic confirmation produces dispositions. It does not exercise authority over repository contents.

---

## Implementation Intent

The semantic stage should operate only on candidates produced by Milestone 1.

It should treat false positives as expected, not exceptional. A candidate can be routed conservatively by deterministic classification and then safely marked false positive by semantic confirmation.

The MVP should avoid over-structuring implementation knowledge. Any rationale should support confirmation and review, not become a premature knowledge schema.

---

## Confirmation Outcomes

Confirmed non-implementation:

The file is a non-implementation artifact that should be recorded for HITL review.

False positive:

The file should not be treated as non-implementation. Examples include implementation code, machine-required artifacts, sanctioned operational artifacts, or files whose purpose is implementation-required despite appearing ambiguous.

Uncertain:

The agent cannot responsibly decide. The file should be recorded for review or later resolution without inventing certainty.

---

## Acceptance Criteria

Milestone 2 is complete when:

* every routed candidate receives a structured semantic disposition
* false positives are explicit outcomes
* uncertainty is explicit and durable
* semantic output can be consumed by the ledger without interpretation
* semantic confirmation remains separate from retention, deletion, and HITL authority

---

## Dependencies

Depends on Milestone 1 for:

* candidate files
* classification evidence

Depends on Milestone 0 for:

* shared vocabulary
* review-loop ownership

---

## Failure Modes

Avoid:

* replacing candidate confirmation with broad implementation-knowledge analysis
* omitting false positives as first-class outcomes
* forcing uncertain files into confident categories
* letting the agent decide keep/delete behavior
* using semantic confirmation as a commit or publication gate
