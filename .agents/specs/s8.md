# Milestone 8 Deep Dive — EvalRoadmap Workflow Implementation

# Purpose

Milestone 8 introduces the EvalRoadmap workflow as a first-class workflow implemented entirely on the Canonical Transition Runtime and Workflow Orchestration Architecture.

Unlike TraditionalRoadmap, which begins with strategic roadmap artifacts, EvalRoadmap begins with evaluation intent.

Its purpose is to progressively transform evaluation intent into a validated implementation-ready roadmap product.

It is **not** a special-case implementation.

It is simply another workflow.

The orchestration runtime must not distinguish it from any other workflow beyond its declarative definition.

---

# Objectives

Implement EvalRoadmap so that:

* it becomes a first-class workflow identity;
* every stage executes through the Canonical Transition Runtime;
* every transition is declarative;
* dependency ordering is explicit;
* outputs are validated before downstream progression;
* the workflow converges into the exact same Plan entry contract as TraditionalRoadmap.

---

# Non-Goals

Milestone 8 does **not**:

* redesign evaluation methodology
* redesign evaluation prompts
* redesign hypothesis generation
* redesign dependency inventory
* redesign DAG generation
* redesign roadmap generation
* redesign milestone generation
* implement transition concurrency
* implement execution loop-back

Only orchestration changes.

---

# Guiding Principles

## Evaluation is a workflow

EvalRoadmap is not a mode inside TraditionalRoadmap.

It is a separate workflow identity.

---

## Dependency-driven

# Canonical Dependency Semantics

Dependencies determine:

transition eligibility.

They do not determine

execution strategy.

Current implementation:

serial.

Future implementation may introduce:

* concurrent dependency branches;
* join transitions;
* deterministic merges.

No runtime redesign should be required when concurrency is later introduced.

---

# Downstream Convergence

One of the primary architectural goals of this milestone is eliminating downstream branching.

Both roadmap workflows terminate with:

```text
Prepared Epic
        ↓
Milestone Specification Set
        ↓
Plan Entry Gate
```

Everything after that point is identical.

Plan,

Execute,

and future workflows

must remain completely unaware of which roadmap workflow produced those products.

---

# Success Criteria

Milestone 8 is complete when:

* EvalRoadmap becomes a first-class workflow identity
* it executes entirely through the Canonical Transition Runtime
* stages represent evaluation semantics
* transitions are declarative
* dependency ordering is explicit
* products are semantic
* runtime owns orchestration
* repository resolution fully supports EvalRoadmap
* default invocation selects EvalRoadmap when evaluation artifacts exist
* `--eval` forces chained execution
* `looprelay eval` remains bounded
* downstream convergence into Plan is identical to TraditionalRoadmap
* no orchestration branching exists after workflow selection
* EvalRoadmap is architecturally independent from TraditionalRoadmap while sharing the same runtime

At completion, the architecture supports two independent roadmap workflows that converge seamlessly into the same Plan and Execute workflows, completing the canonical workflow family. The final milestone can then focus exclusively on unifying the public CLI, retiring legacy orchestration paths, and certifying the entire orchestration architecture as the new authoritative execution model.
