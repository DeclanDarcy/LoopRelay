# Milestone 3 Deep Dive - Non-Implementation Review Ledger

## Purpose

Milestone 3 makes candidate confirmation durable.

The ledger records confirmed non-implementation files for HITL review, records false positives separately, and lets LoopRelay skip identification work for files it has already understood.

---

## Objective

Persist semantic dispositions from Milestone 2 so the review loop can accumulate state across executions and avoid repeated identification.

This milestone answers:

> What has already been identified, and what still needs review or resolution?

---

## Responsibilities

Milestone 3 owns:

* repository-local ledger persistence
* recording confirmed non-implementation files
* recording false positives separately
* recording uncertain candidates
* preserving enough provenance to explain each entry
* skipping semantic confirmation for already identified files when the ledger entry remains valid
* exposing ledger state to synthesis and HITL review

---

## Boundaries

Milestone 3 does not own:

* deterministic classification
* semantic candidate confirmation
* insight extraction
* HITL decisions
* deletion or retention
* prompt guidance
* repository acceptance
* policy versioning beyond what is needed to understand entry validity

The ledger is review state and provenance. It is not a repository knowledge database or acceptance ledger.

---

## Implementation Intent

The ledger should be practical and durable enough for the post-execution epic loop.

It should support the HITL-requested behavior:

```text
record each non-implementation file in ledger
record false positives separately
use ledger to skip identification of already identified files
```

Identity should be strong enough to avoid obvious duplicate work while avoiding speculative replay or repository-evolution machinery.

---

## Ledger States

The ledger should clearly distinguish:

* confirmed non-implementation
* false positive
* uncertain
* resolved by HITL

False positives must not be merged into confirmed non-implementation review state.

---

## Acceptance Criteria

Milestone 3 is complete when:

* confirmed non-implementation files are durably recorded
* false positives are durably recorded separately
* uncertain candidates are represented without forcing a decision
* already identified files can skip semantic confirmation when appropriate
* later synthesis and HITL review can consume ledger state directly

---

## Dependencies

Depends on Milestone 2 for:

* structured semantic dispositions

Depends on Milestone 1 for:

* changed-file and classification provenance

---

## Failure Modes

Avoid:

* treating the ledger as a general repository governance system
* hiding false positives inside generic uncertainty
* requiring repeated semantic confirmation for unchanged known files
* making ledger entries decide file deletion or retention
* implementing speculative invalidation, replay, or certification features
