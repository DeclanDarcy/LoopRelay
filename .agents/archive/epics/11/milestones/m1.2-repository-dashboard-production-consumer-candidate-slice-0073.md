Date: 2026-06-27

## Objective

Emit the first generated repository-dashboard production-consumer candidate from governed schema metadata and verify it against the existing manual TypeScript compatibility wrapper before any production import migration.

## Capability

- Milestone: M1.2 Build the Generated Contract Ecosystem.

## Changes

- Added `RepositoryDashboardConsumerCandidateProjection` and related candidate types to the generated repository-dashboard TypeScript artifact.
- Kept raw observed aliases, including `RepositoryDashboardGeneratedProjection`, as fixture-observed evidence-only aliases.
- Extended governed field metadata with primitive type support for fixture-null fields whose production candidate needs a non-null primitive union, such as provider process id.
- Expanded repository-dashboard governed metadata for execution summary and execution history nullability, execution session state enum domains, identity roles, date/time and duration formats, and repository-relative paths.
- Added structural equivalence verification between `RepositoryDashboardConsumerCandidateProjection` and the manual `RepositoryDashboardProjection` compatibility wrapper.
- Added separate semantic compatibility verification for representative enum domains, nullable summary references, and decision-session status/nullability.
- Regenerated:
  - `tests/CommandCenter.Backend.Tests/ContractFixtures/repository-dashboard.contract-ir.json`
  - `src/CommandCenter.UI/src/contracts/generated/repository-dashboard.generated.ts`
  - `tests/CommandCenter.Backend.Tests/ContractFixtures/repository-dashboard.generated-artifact-freshness.json`
- Updated M1.2 milestone state and durable contract/capability/mechanism documentation.

## Architectural Claim

The generated repository-dashboard production-consumer candidate can represent the manual compatibility wrapper's current production shape without making the generated artifact a production import source. Candidate generation is now a separate verified step between governed schema metadata and consumer migration.

## Known Limits

- The candidate is generated from hand-authored pilot metadata in test support, not yet from a broader backend schema reflection mechanism.
- Optional-by-contract remains unclaimed; candidate properties remain required unless explicitly authorized as optional in future metadata.
- Semantic verification covers representative domains and nullability; it is not yet a full enum-domain equivalence engine.
- Production import migration and compatibility wrapper retirement are still pending.
