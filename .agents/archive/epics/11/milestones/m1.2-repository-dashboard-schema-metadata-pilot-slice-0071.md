# M1.2 Repository Dashboard Schema Metadata Pilot Slice 0071

Date: 2026-06-27

## Objective

Add the first governed schema metadata pilot for `repository-dashboard` so production TypeScript migration blockers are represented as explicit contract facts rather than inferred from the single Oracle fixture.

## Capability

- Milestone: M1.2 Build the Generated Contract Ecosystem.
- Capability: governed schema/nullability/semantic metadata between raw observed aliases and generated production consumer types.

## Changes

- Extended `ContractGenerationIr` with field metadata for presence, nullability, semantic domain, enum values, identity role, array ordering, string format, and source.
- Added `RepositoryDashboardGenerationMetadata` as the narrow pilot source for selected repository-dashboard fields.
- Updated the repository-dashboard generation pipeline to include metadata in:
  - `tests/CommandCenter.Backend.Tests/ContractFixtures/repository-dashboard.contract-ir.json`
  - `src/CommandCenter.UI/src/contracts/generated/repository-dashboard.generated.ts`
- Updated the generated artifact freshness manifest for the regenerated TypeScript contract artifact.
- Added a focused regression proving representative metadata exists for enum domain, nullable summary field, opaque identity, and array ordering.
- Updated durable contract documentation and M1.2 capability/mechanism references.

## Architectural Claim

The generator now has a governed place to carry schema facts that fixtures cannot prove alone. The generated TypeScript artifact still exposes raw observed aliases only; the new metadata is not a production consumer migration and does not retire `src/CommandCenter.UI/src/types/repositories.ts`.

## Compatibility Impact

- No production UI import changed.
- No Rust shell mirror changed.
- No backend response shape changed.
- No manual TypeScript compatibility wrapper was retired.

## Known Limits

- Metadata is pilot-scoped and hand-authored in the generator test support, not yet derived from a broader backend schema reflection mechanism.
- Optional-by-contract fields are not modeled beyond the enum shape because the current selected repository-dashboard fields are treated as required serialized fields.
- Generated production consumer types are still pending; raw generated aliases remain evidence-only.
- Metadata coverage is selected for the known migration blockers and is not yet a complete repository-dashboard schema.

## Rollback

Remove the metadata from `ContractGenerationIr`, regenerate the repository-dashboard generated artifacts through the update path, restore the prior freshness manifest hash, and keep the Slice 0070 generated TypeScript consumer policy as the active production-migration gate.

## Next Slice

Use the governed metadata to emit a separate generated production-consumer candidate for repository-dashboard while keeping raw observed aliases and compatibility wrappers distinct. The candidate should be verified against the manual compatibility wrapper before any production import migration.
