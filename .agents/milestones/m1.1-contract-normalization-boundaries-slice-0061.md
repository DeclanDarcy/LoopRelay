# M1.1 Contract Normalization and Boundaries Slice 0061

## Objective

Extend the M1.1 canonical contract model with contract stability, normalization rules, and boundary semantics before generated artifacts or consumer migrations are introduced.

## Owning Capability

Canonical contract model.

## Architecture Impact

This slice adds durable M1.1 definitions for:

- contract stability classes, separating identity-bearing shape and semantics from additive fields, observational metadata, intentionally unstable values, compatibility transitional fields, and breaking changes;
- normalization rules for identifiers, names, enums, dates and times, optional values, collections, metadata, ordering, evidence, diagnostics, compatibility fields, error envelopes, and streams;
- boundary semantics for domain-to-projection, projection-to-contract, contract-to-transport, transport-to-resource, resource-to-controller, controller-to-workspace, and workspace-to-presentation transitions;
- request, response, error, and stream boundary rules.

The slice keeps M1.1 model-first. It does not add generation, change fixtures, alter endpoint behavior, migrate Rust mirrors, migrate TypeScript types, or expand dev Tauri mock coverage.

## Compatibility Impact

No runtime compatibility surface changed. Existing compatibility obligations remain:

- Rust repository dashboard and workspace mirrors still have known `decisionSessionSummary` drift.
- Manual TypeScript repository and workflow types remain verified compatibility artifacts, not contract authority.
- Dev Tauri mock repository dashboard and workspace payloads remain verified compatibility consumers.
- Dev mock workflow response coverage remains absent.

The new stability and normalization rules define how those obligations must be evaluated before M1.2 generation and M1.3 transport work.

## Governance

This slice follows the active decision to continue M1.1 with normalization and boundary semantics before M1.2 generation. It treats stability as distinct from versioning and compatibility, so later Oracle, fixture, generator, compatibility, and regression mechanisms do not treat every JSON value change as a contract identity change.
