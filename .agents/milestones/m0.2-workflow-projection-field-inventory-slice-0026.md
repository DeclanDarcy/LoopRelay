# Milestone 0.2 Slice 0026: Workflow Projection Field Inventory

Date: 2026-06-26

## Objective

Begin workflow projection Contract Oracle coverage with gated field inventory only. This slice identifies the primary workflow projection contract, producers, consumers, parallel representations, compatibility obligations, request boundaries, fixture candidate fields, and semantic lifecycle fields before any golden fixture is added.

## Scope

Contract identity: Workflow projection.

Primary endpoint: `GET /api/repositories/{repositoryId}/workflow`.

Projection: `WorkflowInstance`.

Authority: backend-owned workflow projection shape after backend HTTP JSON serialization.

Out of scope for this slice:

- golden workflow fixture,
- workflow consumer-verification tests,
- workflow artifact freshness manifest,
- workflow request-boundary tests,
- sibling workflow endpoint fixtures for diagnostics, history, gates, recovery, continuation, preparation, health, reports, and certification.

## Evidence Observed

- `src/CommandCenter.Backend/Endpoints/WorkflowEndpoints.cs` maps `GET /api/repositories/{repositoryId:guid}/workflow` to `IWorkflowProjectionService.ProjectAsync`.
- The same endpoint file maps 27 sibling workflow endpoints that expose related diagnostics, transitions, gates, recovery, execution, handoff, decisions, operational-context, Git, continuation, preparation, health, report, and certification contracts.
- `src/CommandCenter.Workflow/Models/WorkflowInstance.cs` defines the primary workflow projection contract.
- `src/CommandCenter.Workflow/Services/WorkflowProjectionService.cs` composes the projection from execution, handoff, decision, operational-context, Git, decision-session, gate, timeline, and state-machine evidence.
- `src/CommandCenter.Workflow/Models/*.cs` defines nested workflow projections, diagnostics, gates, transitions, completion, timeline, and decision-session governance shape.
- `src/CommandCenter.Shell/src/main.rs` exposes `get_workflow_projection(repository_id: String) -> Result<Value, String>` and relays the backend response through `serde_json::Value` rather than a Rust `WorkflowInstance` response mirror.
- `src/CommandCenter.UI/src/api/workflow.ts` maps `getWorkflowProjection(repositoryId)` to `get_workflow_projection` with only `{ repositoryId }`.
- `src/CommandCenter.UI/src/types/workflow.ts` manually mirrors `WorkflowInstance` and related workflow contract types.
- `src/CommandCenter.UI/src/devTauriMock.ts` does not currently expose an explicit `get_workflow_projection` mock command handler.
- `src/CommandCenter.UI/src/hooks/useWorkflowProjection.ts`, workflow feature panels, workspace rail, execution rail, governance workspace, selected repository summary, and shell header consume `WorkflowInstance`.
- `tests/CommandCenter.Backend.Tests/WorkflowProjectionServiceTests.cs` contains broad backend projection behavior coverage but no Oracle golden fixture for serialized workflow projection shape.
- `src/CommandCenter.UI/src/test/characterization/transport.test.ts` includes a narrow workflow command request/response characterization for the TypeScript API wrapper.

## Fixture Candidate Requirements

The first workflow fixture should cover:

- one non-empty `repositoryId` GUID string,
- string enum serialization for stage, progress, gate, execution, handoff, decision, operational-context, Git, gate, blocking-condition, timeline, and preparation-like values reachable from the primary projection,
- explicit nulls for optional ids, timestamps, failure, gate resolution, blocking condition, completion artifact, and nullable decision-session values,
- non-empty arrays for timeline, open/satisfied gates, valid/blocked transitions, diagnostics, and evidence,
- at least one empty array in nested diagnostics to prove empty collection preservation,
- nested `decisionSession` either populated or explicitly null, with the chosen state documented,
- backend-owned eligibility booleans and governance/health/diagnostic fields without UI reinterpretation,
- ordered arrays whose ordering is semantically owned by workflow projection or gate/timeline services.

## Compatibility Findings

- The workflow family has 27 sibling endpoint contracts. They must not be silently certified by the primary `WorkflowInstance` fixture.
