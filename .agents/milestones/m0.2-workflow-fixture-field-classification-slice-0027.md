## Classification Vocabulary

| Classification | Meaning for fixture review |
| --- | --- |
| Semantic authority | Backend-owned lifecycle, eligibility, status, legality, health, recovery, governance, or diagnostic meaning that downstream consumers must render or sequence from without recomputing. |
| Structural metadata | Identity, timestamp, source, artifact, fingerprint, ordering, or correlation data that gives semantic fields traceability and stable serialized shape. |
| Compatibility field | Duplicated, denormalized, transitional, or convenience field whose presence is currently required by consumers and must derive from authoritative structure. |
| Diagnostic field | Backend-owned evidence, reasoning, conflict, missing-evidence, warning, or error details used to explain a projection decision. |
| Derived presentation helper | Backend-provided summary text or human action text intended for direct UI display, not for downstream semantic inference. |

## Fixture Acceptance Gates

The workflow golden fixture remains blocked until the fixture evidence demonstrates:

- each top-level `WorkflowInstance` field above is present in the candidate fixture or explicitly excluded by a decision record,
- at least one explicit null is present for nullable object fields such as `executionFailure`, transition `gateResolution`, transition `blockingCondition`, completion artifact, nullable ids, timestamps, or `decisionSession`,
- every diagnostics group has required array fields represented, with at least one empty array and at least one non-empty array across the fixture,
- backend-owned eligibility booleans are represented as booleans, not inferred from status strings,
- ordered arrays selected for `timeline`, transitions, and gates have order treated as semantically owned by backend workflow services,
- flattened compatibility fields are compared against their owning nested/source fields in fixture evidence before consumer migration work begins,
- sibling workflow endpoints remain outside the fixture claim.
