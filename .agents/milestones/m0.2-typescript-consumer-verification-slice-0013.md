# M0.2 TypeScript Consumer Verification Slice 0013

Date: 2026-06-26

## Objective

Extend repository dashboard consumer verification from a single Rust downstream mirror to a multi-consumer mechanism by adding manual TypeScript type verification against the Oracle fixture.

## Capability

Canonical contract Oracle.

## New Evidence

- `ContractConsumerVerificationTests` now includes a TypeScript consumer shape provider.
- The TypeScript provider parses exported type aliases under `src/CommandCenter.UI/src/types`.
- The provider resolves aliases across the shared type folder, including imported dashboard dependencies such as `ExecutionReadiness`, `RepositoryExecutionState`, and `ExecutionSessionSummary`.
- The provider treats TypeScript string-literal unions as string-valued contract shapes.
- The provider unwraps `T | null` nullable unions and compares array item shape when the backend fixture contains representative items.
- `RepositoryDashboardTypeScriptTypeMatchesGoldenFixture` verifies the manual `RepositoryDashboardProjection` has no missing, extra, or value-kind drift against `repository-dashboard.golden.json`.
- `RepositoryDashboardTypeScriptTypeRecursivelyVerifiesImportedNestedShape` verifies imported execution summary aliases and nested decision-session summary arrays participate in the shared recursive comparison.

## Architectural Impact

TypeScript remains a downstream compatibility consumer. This slice does not promote manual TypeScript types to contract authority and does not change backend serialization truth. The Oracle remains backend JSON serialization observed through the golden fixture.

The consumer verification mechanism now has two shape providers feeding the same recursive comparison model:

- Rust shell mirror provider.
- TypeScript manual type provider.

## Current Findings

- TypeScript `RepositoryDashboardProjection` currently matches the repository dashboard Oracle fixture shape.
