# Milestone 0.2 Slice 0023: Repository Workspace Request Boundary

## Objective

Extend the repository workspace Oracle pilot with request-boundary verification for the primary workspace GET path, reusing the repository dashboard request-boundary mechanism.

## New Protection

- Added backend route verification for `GET /api/repositories/{repositoryId:guid}/workspace`.
- Added Rust command verification for `get_repository_workspace(repository_id: String)`.
- Added TypeScript API wrapper verification for `getRepositoryWorkspace(repositoryId: string)`.

## Mechanism Reuse

This slice reused the existing `ContractRequestBoundaryTests` source-inspection and endpoint-metadata pattern.

The workspace verifier is the first non-empty request-boundary pilot. It proves one required route/command argument is preserved across:

- backend route metadata,
- Rust Tauri command arguments and GET forwarding,
- TypeScript command argument object construction.

It does not introduce a general request-contract model, command DTO artifact, query/body compatibility classifier, or generated request-boundary artifact.

## Limits

- Repository workspace local Oracle certification is still pending.
- Coverage is limited to the primary workspace GET path; workspace refresh and artifact rotation request boundaries remain pending.
- Request-boundary verification still does not classify route/query/body compatibility generally.
- Milestone 0.2 remains active and uncertified globally.
- Known Rust shell response mirror drift remains: `RepositoryWorkspaceProjection` omits `decisionSessionSummary`.

## Rollback

Remove the three repository workspace request-boundary tests and revert the documentation updates. The workspace Oracle fixture, response consumer verification, and artifact freshness checks would remain intact, but workspace request-shape drift would no longer be executable.
