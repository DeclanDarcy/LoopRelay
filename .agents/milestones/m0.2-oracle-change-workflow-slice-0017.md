## Architectural Impact

This slice governs how contract drift becomes an accepted baseline. It does not add a new Oracle branch, generator, fixture family, or production contract migration.

The workflow now explicitly covers:

- fixture comparison drift,
- downstream consumer drift,
- artifact freshness drift,
- required change-record fields,
- fixture and artifact action order,
- repository dashboard pilot verifier commands,
- evidence and rollback requirements.

## Current Coverage

The workflow is procedural and applies first to the repository dashboard pilot.

It preserves the three-branch Oracle model:

- fixture comparison observes backend serialization against accepted Oracle truth,
- consumer verification checks downstream conformance to the Oracle fixture,
- artifact freshness checks whether tracked artifacts moved with their Oracle source.

## Protected Failure Modes

- Updating fixtures before proving the backend change is authoritative.
- Treating consumer drift as a reason to weaken the Oracle fixture.
- Accepting a freshness baseline without classifying stale artifacts, unexpected manual artifact edits, or missing artifacts.
- Moving a contract baseline without rollback evidence.
