## Architectural Property Check

The first two pilots proved the Oracle lifecycle across repository read-model contracts. The workflow pilot added the next highest-value property: a semantically dense projection with nested lifecycle, diagnostics, transitions, gates, backend-owned eligibility booleans, ordered arrays, flattened compatibility fields, and explicit nullable `decisionSession` state.

No substantial new Oracle property was discovered that requires a fourth contract family before moving toward milestone-level certification. A fourth family would add breadth, but the next highest leverage gap is not another fixture by default. The stronger gap is milestone-level synthesis: identify exactly which Milestone 0.2 exit criteria can now be certified, which remain partial, and which known gaps should become explicit certification limitations or next-milestone dependencies.

## Decision Lifecycle Eligibility Deferred

Decision lifecycle eligibility remains the preferred fourth family if milestone certification review finds that backend-owned eligibility semantics are still underrepresented. It should not be added automatically as breadth-only coverage, because workflow already exercises backend-owned eligibility fields and downstream semantic drift risk.

Error envelope remains important for runtime and transport architecture, but it primarily exercises failure representation and preservation rather than semantic authority. It is better evaluated when Milestone 0.2 certification or Milestone 1.3 passive transport needs failure-envelope evidence.

## Known Limits

- Repeatability is proven across three pilots only: repository dashboard, repository workspace, and primary workflow projection.
- Workflow dev mock handler verification remains absent and accepted only for the initial workflow pilot.
- No populated `decisionSession` workflow fixture variant exists.
- Sibling workflow endpoints are not certified.
- Most endpoint families still lack field-level inventories and golden fixtures.
- TypeScript artifact freshness still protects Phase 0 verified manual contract artifacts, not generated Milestone 1.2 output.
- Semantic reinterpretation checks, deterministic generation, fixture update automation, mechanical versioning, stream fixtures, error-envelope fixtures, non-empty command-body verification, and broad dependency graph coverage remain pending.
