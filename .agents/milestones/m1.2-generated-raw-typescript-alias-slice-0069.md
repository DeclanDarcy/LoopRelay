# M1.2 Generated Raw TypeScript Alias Slice 0069

## Objective

Expand the repository dashboard generated TypeScript artifact from metadata-only output to raw generated contract aliases while keeping production UI consumers on the existing compatibility wrapper.

## Scope

- Pilot contract family: `repository-dashboard`.
- Source authority: `tests/CommandCenter.Backend.Tests/ContractFixtures/repository-dashboard.golden.json`.
- Generated artifact: `src/CommandCenter.UI/src/contracts/generated/repository-dashboard.generated.ts`.
- Verification: generated alias shape participates in the existing TypeScript consumer verifier.

## Implementation

- Extended `ContractTypeScriptMetadataGenerator` to emit named raw TypeScript aliases from the canonical generation IR.
- Added `RepositoryDashboardGeneratedContract` and `RepositoryDashboardGeneratedProjection` aliases plus generated nested object aliases.
- Updated the generated artifact freshness manifest identity from metadata-only to generated contract output.
- Extended `TypeScriptContractShapeProvider` to read generated contract directories and resolve `null`-only generated fields for fixture comparison.
- Added `RepositoryDashboardGeneratedTypeScriptAliasMatchesGoldenFixture`.

## Architectural Claim

The slice proves that the M1.2 generator can produce deterministic raw TypeScript contract aliases from the accepted IR and that the generated alias shape matches the Oracle fixture through the same consumer-verification mechanism used for manual TypeScript types.

## Compatibility

No production UI consumer was migrated. A direct attempt to export the raw generated projection through `src/CommandCenter.UI/src/types/repositories.ts` was rejected because the current IR is fixture-observed rather than schema-complete: enum-like fields widen to `string`, and observed nullability narrows fields to `null` or non-null based on the single fixture. The manual `RepositoryDashboardProjection` compatibility wrapper therefore remains the production type until M1.2 adds governed schema/nullability and semantic alias mapping.

## Rollback

Remove the generated raw aliases from `ContractTypeScriptMetadataGenerator`, regenerate `repository-dashboard.generated.ts`, restore the generated freshness manifest to the metadata-only artifact identity, and remove `RepositoryDashboardGeneratedTypeScriptAliasMatchesGoldenFixture`. Production UI behavior is unaffected because no product import was migrated.

## Known Limits

- Generated raw aliases are fixture-observed, not schema-complete.
- Generated raw aliases are verified but not production-facing.
- Manual dashboard TypeScript types remain compatibility consumers.
- Full consumer migration requires explicit nullable union modeling, semantic enum mapping, and compatibility-wrapper retirement criteria.
