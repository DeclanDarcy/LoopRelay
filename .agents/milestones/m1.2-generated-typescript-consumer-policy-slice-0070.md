# M1.2 Generated TypeScript Consumer Policy Slice 0070

## Objective

Define the governed schema/nullability and semantic alias policy required before generated TypeScript aliases can become production consumer types.

## Scope

- Milestone: 1.2 Generated contract ecosystem.
- Pilot contract family: `repository-dashboard`.
- Durable documentation: `docs/contracts.md`.
- Active generated artifact status: raw generated aliases remain evidence-only.

## Implementation

- Added `Generated TypeScript Consumer Policy` to `docs/contracts.md`.
- Classified generated TypeScript output into raw observed aliases, production consumer types, and compatibility wrappers.
- Defined required field-level policy for structural JSON kind, nullable-by-contract, omitted-by-contract, semantic enum domains, opaque identity, arbitrary text, array ordering, empty collections, and date/time or duration strings.
- Recorded the approved repository dashboard migration path:

```text
Raw generated observed alias
  -> governed schema/nullability/semantic metadata
  -> generated production consumer type
  -> compatibility wrapper alias or adapter
  -> existing production consumers
  -> compatibility wrapper retirement evidence
```

- Updated the M1.2 milestone, capability matrix, and mechanism catalog to reference this policy boundary.

## Architectural Claim

The generated contract ecosystem now has a documented consumer-migration gate: raw generated aliases may verify observed shape, but production TypeScript consumers require explicit schema facts before the manual compatibility wrapper can be retired.

## Compatibility

No production UI import changed. `src/CommandCenter.UI/src/types/repositories.ts` remains the compatibility wrapper and visible semantic-strengthening location for repository dashboard consumers.
